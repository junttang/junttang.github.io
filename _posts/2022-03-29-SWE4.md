## 객체지향 SW 개발 방법론의 발전
&nbsp; **과거부터 다양한 객체지향적 SW개발 방법론들이 도입**되었다.
- Object Modeling
- Use Cases
- Statechart
- Sequence Diagram
- Object Collaboration Diagram
- Concurrent, Distributed & Real-Time Design
  - CODARTS (COcurrent Design Approach for Real Time Systems)
  
**~> 이들이 발전하여 최종적으로 아래의 COMET 방법론**이 나온 것이다.

<br>

## Unified Modeling Language (UML)
&nbsp; **'통합된 모델링 언어(Unified Modeling Language)'는 객체지향 SW개발에 쓰이는 언어(표기법)이다.** (프로그래밍 언어를 말하는 것이 아니다)
**~> 앞선 여러 방법론은 제각기 따로 노는 개념들이었는데, 이들을 통합한 것**이다.

> _**UML은 여러 객체 지향 방법론의 '표준화' 결과이다. 타인이 자신의 설계를 이해할 수 있도록!**_

- **객체지향 개발의 Standardized Notation**이다.
- OMT, Booch, Use Case와 같은 **여러 방법론을 통합**하였다.
- 객체지향 개발 과정을 **Graphical하게 표현**하는 언어이다.
- **OMG**(Object Management Group)에서 **공인**하였다.
- 현재 2.0 Version까지 나왔다.

<br>

## COMET
&nbsp; **COMET은 'Concurrent Object Modeling and architectural design mEThod'의 약자이다. 즉, '동시적 객체 모델링 및 구조적 설계 방법'**이다.

> _**COMET은 객체 지향 분석 및 설계 기법**이다._
_**COMET = UML + Method : UML이란 Notation을 가지고 어떻게 분석하고 설계할지에 대한 방법론이다.**_

- **COMET의 특징**
  - **동시적 객체 지향 설계를 위한 단계와 가이드라인을 제시**한다.
  - **Concurrent, Distributed & Real-Time 응용을 설계하는 기준들을 제시**한다.
  
**ex)** 도서관 관리 SW는 데이터의 정리와 탐색 기능이 중요하다. 반면에, 가정용 로봇 SW는 데이터보다는, 특정 명령에 대한 즉각적 반응과 문제 상황 대처 능력, 실시간 문제 대응 능력(Concurrent & Distributed & Real-Time) 등이 중요하다.

**~>** COMET은 이 중 후자의 예시와 같은 응용에 집중한 방법론이다.

<br>

## COMET 객체지향 SW 생명주기 모델
![](https://images.velog.io/images/junttang/post/30d5bbea-8dd2-4691-9a08-56031e4758e7/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

- **Waterfall Model과 유사**하다.

- **[요구사항 -> 분석 -> 설계 -> 점진적 SW 구현 -> 점진적 SW 통합 -> 시스템 테스트 ]의 흐름**으로 나아간다.
  - **요구사항에서 분석으로 나아가는 과정에서, 철저한 프로토타이핑 과정**이 수행된다.
  - **점진적 SW 구현에서 테스트로 나아가는 과정에서, 점진적인 프로토타이핑 과정**이 수행된다.
  
**※ Integration : 만든 모듈들을 합치는 단계**이다.

<br>

### 1. Requirements Modeling
&nbsp; COMET 생명주기 모델에서 **Requirements Modeling은 곧 'Use Case Modeling'**이다.

- **Use Case와 Actor의 관점에서 SW 기능적 요구사항들을 정립**한다.
  - **Use Case와 Actor는 SW가 어떤 기능을 제공해야하는지를 표현**한다.
  - Use Case는 Diagram인데, 이에 대한 상세 개념은 추후 설명한다.
- **고객의 요구사항을 UML의 Use Case와 Actor라는 표현기법으로 표현**하는것이다. (그래서 Modeling이라 하는 것)

<br>

### 2. Analysis Modeling
&nbsp; 요구사항을 정립했으니, 이제 분석 단계에 들어간다.

&nbsp; 우리가 Analysis를 왜 하는가? 그렇다. "대상을 이해하기 위해서" 분석한다. 이때, '무언가를 이해'하기 위해선, 다각도로 바라봐야한다. 여기서도 똑같다.

- **Static Modeling**
  - **Class 단위로 요구사항을 분석**한다.
  - **Class 간의 구조적 관계를 정립**한다.
  - 이를 **Class Diagram에 표현**한다.
    ![](https://images.velog.io/images/junttang/post/199dccb5-fd9e-4562-9d97-6518cf8862ac/bsds.png)

- **Dynamic Modeling**
  - **(Object의) 행위 단위로 요구사항을 분석**한다.
  - **UML의 Communication Diagram을 기반으로 표현**한다.
  - **State Machine Modeling** with Statecharts : **상태 기계(State Dependent Control Object)들을 상태도로 표현**한다.
  ![](https://images.velog.io/images/junttang/post/48a11101-e6aa-4f86-9e4b-6af7ffdd050f/basdsadasd.png)
  - **Object Interaction Modeling** : **Collaboration Diagram이나 Sequence Diagram으로 특정 Use Case에서의 Object들의 행위를 표현**한다.
  ![](https://images.velog.io/images/junttang/post/7f1b038c-979b-421f-b383-2dce7c31d708/whatis-collaboration_diagram.png)
  
<br>

### 3. Archtectural Design
&nbsp; 설계 단계이다.

> _**Design의 목적은 Solution을 만드는 것이다. 모듈을 어떻게 만들지와 관련된다.**_

- **SW 모델, Task 단위의 설계**가 이루어진다.

- **전반적인 SW 구조를 개발**한다.
  - 2번 분석 단계에서 만든 **Collaboration Diagram을 토대로 Synthesize** 과정을 거친다.
  
- **기반이 되는 Subsystem 요소들을 설계**한다.
![](https://images.velog.io/images/junttang/post/09a90ecf-7f89-4177-aac2-437b425984ee/2.png)

- **동시 & 분산 & Real-Time 응용**
  - **객체지향** 개념 도입 : 디테일 정보 숨기기, 클래스로 표현하기, 상속하기
  - **Concurrent Tasking**
    - **Active 객체, Passive 객체**
    - **Centralized 제어 방식, Distributed 제어 방식**
    - **Message 통신 방식 : 동기식, 비동기식, 그룹식 등**
    
- **Real-Time Design의 성능을 분석**한다.
  
<br>

### 4. Incremental Development & Integration
&nbsp; **설계를 토대로 점진적으로 SW 개발을 수행**한다. (실제 프로그래밍 부분)

- **SW의 구조적 설계를 마치고 나서** 이루어진다.

- 점진적으로 수행한다.
  - **Use Case를 바탕으로** 시스템의 부분을 설계한다.
  - 구체적으로 디자인하고, 클래스들을 테스트하고 등등
  
- **점진적으로 통합**한다.
  - 쉽게 말해서, 코딩 끝나면 합친다!
  - **각 시스템 요소들을 테스팅한다. Use Case 바탕으로!**
  - **각 Use Case에 대한 Test Case들**을 만든다.
  - **Use Case의 객체 간의 인터페이스들을 테스트**한다.
  
**※ White Box Testing : 하나하나 모듈 단위로 모듈을 테스트하는 것을 말한다. 각각의 Logic을 체크하는 과정이다.**

<br>

### 5. System Testing

- **Functional Testing of System**

- **Black Box Testing : 모듈을 다 모아놓고, 통합해놓고 전체를 테스팅하는 것을 말한다. 안에서 어떤 흐름으로 하는지엔 관심없이, 오로지 Input에 따른 Output을 테스트한다고 해서 Black Box라는 이름이 붙은 것이다. 반대가 바로 위의 White Box!**

- 각 단계마다 시스템을 테스트한다.

- **개별적인(독립적인) Test Team을 따로 둔다.**
  - 보통, 기업에서는 테스트 팀과 개발 팀을 별도로 둔다. **시야를 넓게 해서 평가하기 위함**이다.
    - System을 분해(Break)해서 **철저하게, 시스테믹하게 평가한다. 소비자에게 우수한 제품을 납품하기 위해서** 말이다.
    
- 테스트가 끝나면 SW제품을 납품하게 된다.
