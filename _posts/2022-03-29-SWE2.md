## Software Process
&nbsp; **소프트웨어 프로세스 : 소프트웨어 제품 개발 과정에서 밟는 일련의 절차
~> 'Software Lifecycle'과 동일한 개념**이라고 봐도 무방하다.

&nbsp; 소프트웨어의 Lifetime은 하루 단위부터 년 단위까지 다양하다.

&nbsp; 이러한 Software Lifetime Process는 아래와 같은 요소들로 구성된다.
- People : 소비자와 개발자
- Overall Process : 전체적인 과정
- Intermediate Product : 과정 중간의 결과(제품)
- Stages of the Process : 과정의 중간 단계들

<br>

**※ 소프트웨어 개발의 전형적인 단계**
**1) Requirement** : 요구사항 (정의) 단계 ~> 의외로 현실에서 간단한 단계가 아니다.
**2) Analysis** : (요구사항) 분석 단계 ~> 개발자가 요구사항, 의도를 이해하기 위해 수행
**3) Design** : SW 설계(Architecture) ~> Program의 구조에 대해 Solution을 만드는 일
**4) Implementation** : SW 구현
**5) Testing** : SW 테스트
**6) Maintenance** : 유지보수 ~> 제품을 납품하고 끝나는 것이 아니라, 유지보수도 중요

**여담)** **Requirements 단계가 의외로 어려운 이유** : 고객이 원하는 요구사항과 관련된 Field에 개발자가 익숙치 않을 수 있고, 또 반대로 고객이 SW 시스템에 익숙치 않을 수 있기 때문에 이 과정은 결코 말처럼 간단한 단계가 아니다.

<br>

## Software Lifecycle Model (SW Process Model)
&nbsp; 상기한 'SW 개발의 전형적 단계'를 포함해, SW 개발과 관련한 **다양한 프로세스 모델**이 존재한다. 이들의 특징과 목표에 대해서 알아보자.

### (1) Waterfall Model

![](https://images.velog.io/images/junttang/post/e982ddc6-666b-4c31-9144-e9ca301d03f3/350px-Waterfall_model.svg.png)

- 우리가 생각하는 가장 **전형적**인 모델
- 최초 시도에서, 요구사항을 명확하고 "완벽하게", "Consistently하게" 구체화한다.
- 최초 시도에서, 시스템을 정확하고 완벽하게 설계한다.
- 최초 시도에서, 프로그램 인터페이스와 내부 설계를 완벽하고 정확하게 작성한다.
- **하나의 큰 단계에서 시스템 요소들을 통합**한다**(Large Integration)**.
- 모든 단계를 마친 후, 가장 마지막 단계에서 테스팅 과정을 거친다.

> _**Waterfall 모델의 특징은, 유지보수 단계 이전까지는 모두 "첫 시도에 완벽하게"한다는 것이다.**_

> **Requirements Definition -> SW Design -> Implementation & Unit Testing -> Integration & System Testing -> Operation & Maintenance**

(**유지보수 이전 단계에서는 각 단계가 다음 단계로만 이동이 가능하고, 유지보수에서만 이전 단계로의 Back이 가능**하다.)

<br>

- **장점** : 이론과 관리 측면에서 우수하다.
  - 현재 단계를 완벽하게 수행하고 나서 다음 단계를 수행한다.
  - **"완벽"을 위해, 각 단계에서 'Progress Review'를 진행**한다.
  - **'Product Baseline(제품 개발 과정의 형상, 다이어그램)'을 확립하고 제어하기가 수월**하다.
  
  <br>
  
- **단점** : 중간중간 변화가 있으므로 순서대로 나아가기가 힘들다. **현실적 측면에서 매끄럽게 적용하기가 힘들다.**
  - **최초 단계에서 소비자가 반드시 요구사항을 다 알려주어야 한다.**
  - 개발자는 각자의 영역에서 모두 완벽해야한다**(Expert여야한다.)**
  - **Integration 과정과 Testing 과정이 가장 마지막에 이루어져, 그 규모가 크고, 노력이 많이 필요**하다.
  - 시스템이 마무리되기 전까지는 시스템의 성능에 대한 어떠한 평가, Demonstration도 없다. **즉, 뒤늦게 시스템의 개선점을 알아차릴 확률이 높다.** 따라서 이러한 'End-Item Testing' 상황에서는 결점을 발견하더라도 어느 부분에서 만들어진 결점인지 알기 쉽지 않다.
  - _**"Defects injected first are defected last!"**_
  
<br>

### (2) Incremental Model
&nbsp; **똑같이 단계별로 하되, 각 단계나 각 기능이 완성될 때마다 여러 버전을 만들어 계속해서 피드백을 하는 방식**이다. 전체적인 형상은 Waterfall 모델과 유사하지만, Waterfall의 "완벽하기 전까진 가지 않는다."라는 Rule이 없는 것이라 볼 수 있다.

![](https://images.velog.io/images/junttang/post/2b23c71a-3d08-4986-ab40-b51f77cff6b4/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

- **Design이 일련의 'Prioritized Builds(우선순위화된 과정들)'로 분할**된다.
  - **각각의 빌드는 기존의 단계(Base)들에 우선순위에 따라 기능들을 추가**해간다.
    - 즉, **각 빌드는 '입증된 버전(Demonstrable Version)'**을 만들어낸다.
   

- **가장 중요한 부분은 처음 빌드되고, 이들은 이 빌드가 쌓여감에 따라 가장 많이 테스트되고, 검증된다.**
  
![](https://images.velog.io/images/junttang/post/e120e671-39c5-4925-b489-a36cd4cfb22c/asd.png)

**ex)** Incremental Model의 예시 (Analysis와 Design은 끝마치고, Implementation을 하는 상황)
Version 1 : 입력 기능 추가 ~> Demo1  &nbsp; &nbsp; **-> 이 기능이 가장 많이 검증된다.**
Version 2 : 테이블 추가 ~> Demo2
Version 3 : 에러 메세지 추가 ~> Demo3
Version 4 : 완벽 ~> Deliver to the Customer
...

**~> 이때의 각 버전**

<br>

- **장점** : 아무래도, 각 단계에서 문제를 미리 알 수 있다는 점이 가장 큰 장점
  - **초기와 중간 단계의 과정에 대해 입증할 수 있음**
  - 처음 추가된 요소들이 가장 많은 검증을 받음
  - **이후 버전에 새로운 기능, 변화 등을 추가**할 수 있음
  - **기능과 일정을 절충(Tradeoff)시킬 수 있음.**
    - Development 과정과 Delivery Time을 구분
  - **중간 개발 단계의 버전을 소비자에게 Deliver하여 피드백**을 받을 수 있음.
  
**※ Incremental Delivery** : 중간 단계 버전을 소비자에게 보여주는 것
**~>** 사용자로부터 초기부터 피드백을 받을 수 있음.
**~>** **일정 내에 제품을 전달할 가능성이 높아짐**. 왜냐? 중요한 기능들부터 작업이 들어갈 것이므로.
**~>** **Later Version들에 대한 Systematic Planning을 적용**할 수 있음. 왜냐, 이전에 피드백들을 바탕으로 계획을 세울 수 있으므로.
**~>** 80%의 기능들로 완성한 제품을 보내는 것이, 만들어놓은 기능들은 100%의 기능들이지만 아직 완성을 못한 제품을 보내는것보다는 훨씬 나음.

<br>

### (3) Evolutionary Model
&nbsp; **진화적, 실험적 모델**이라고 하는 이 모델은, "요구사항 자체가 개발자와 요구자 둘 다 **서로 소통이 잘 안되는**, 서로 분야에 대해 잘 모르는, 그러한 상황일 경우에 사용할 수 있는 프로세스 모델이다.
**~> 요구사항에 대한 몇 가지 경우의 사이클을 만들어놓고, 순서대로 각 사이클을 쫙 돌려보면서 알맞는 것이 무엇이었는지를 찾아내는 방법이다. 그래서 '실험적'**이라 하는 것이다.

![](https://images.velog.io/images/junttang/post/6b8f0568-66b0-40c6-ac9b-00fb96c9ad49/asdasdasd.png)

&nbsp; 즉, 브루트포스하게 개발하는 것이다.

&nbsp; 이러한 모델에서 프로세스(사이클)은 아래와 같은 3가지 경우 중 하나에서 끝나게 된다.
**1) 도저히 가망이 없는 경우, 개발을 취소 (Infeasible)
2) 개발 완성
3) 프로세스가 어느정도 해결은 되려는 기미가 보여 Incremental Model로 전환**

**~> _하이리스크_인 프로젝트에서 사용**하곤 한다.

<br>

### (4) Spiral Model

![](https://images.velog.io/images/junttang/post/90950fa2-42a0-43c5-a20c-d22f223a2886/333px-Spiral_model_(Boehm,_1988).svg.png)

&nbsp; 최근엔 잘 사용하지 않는 구형의 방식임.

- **의의(장점)**
  - **'위기 평가(Risk Assessment)' 개념의 도입**
  - **'위기 관리(Risk Management)' 개념의 도입**
  
- **단점**
  - 나선형 반복마다 Massive Iteration & Continuous Assessment : 즉, 반복이 많음.
  - 그래서, 잘 적용하지 못하게 되면 SW 개발 과정 자체가 **소모적**으로 변할 수 있음.
  
<br>

### (5) Unified Software Development Process (통합형 SW 개발) Model : USDP
&nbsp; 말 그대로 **상기한 모델들을 포함해 다양한 개발 모델을 짬뽕**시킨 것이라 볼 수 있다.

&nbsp; **UML을 이용해 프로세스가 진행**된다. (UML로 개발하는 과정에서 나온 방법론임.)

&nbsp; **복잡한, 객체지향 SW 개발 시에 애용**된다.

- **USDP의 3가지 기본 원칙**
  - **Use-Case Driven** (활용 사례 지향)
  - **Architecture-Centric** (설계, 구조 중심)
  - **Incremental & Iterative** (점진적이고 반복적으로)
  
- Phase들이 전체 개발 사이클, 프로세스를 구성한다.
  - **USDP의 Phase : Inception, Elaboration, Construction, Transition**
- Core Workflow들이 작업을 구성한다.
  - **USDP의 Workflow : Requirements, Analysis, Design, Implementation, Test**
  
- Several Interations make up a 'Phase'

![](https://images.velog.io/images/junttang/post/1140be52-1879-4b28-9da8-9ea16c851785/dssdadsad.png)

- **Phase들이 프로세스를 Time에 따라 구성하고, Workflow들이 프로세스를 Task에 따라 구성하는 것이다. 그리고 각 Phase들은 Iteration들로 구성되는 것이다. 이 구분이 중요하다.**

![](https://images.velog.io/images/junttang/post/08ab4169-3dfc-4ead-8039-09e4ca7619b3/asdasdasdasdasd.png)

**=> Phase와 Workflow는 위와 같은 관계**를 가진다.
**1) Inception = 요구사항 + 분석 // 2
2) Elaboration = 요구사항 + 분석 + 설계 // 3
3) Construction = 설계 + 구현 + 테스트 // 3
4) Transition = 구현 + 테스트 // 2
**
<br>

### (6) Extreme Programming Model
&nbsp; **'Lightweight', 'Small', 'Quick', 'Agile', 'Small Iteratons'**란 단어들로 표현할 수 있는 모델로, **소규모의 팀이 빠르게 기민하게 개발을 수행하는 형태**를 의미한다.

&nbsp; **기본은 Incremental Model**이다. 과거엔 개발 과정에서 변화를 주는 것이 어려웠지만, **현대 기술의 향상으로 빠르고 간단한 변화가 가능해져 이런 방식이 흥하기 시작했다.(Motivation)**

- 잠재적으로 추가할 기능을 비용을 고려하여 식별한다. in Requirement 단계에서
~> 즉, **초기 조건들은 모두 비용-이득 분석을 거쳐 선택된 것**들이다.

- 이 모델에서 **Design 단계는 강조되지 않는다.**

- **Implementation 단계에서는 Pair Programming**으로 수행된다.

- **'Ego-less Programming'이 강조**된다. 팀 구성원들간에 서로 자존심이나 '자신의 특별함'을 내세우지 않고 각자 할일을 맡아서 하는 그런 분위기 말이다.

- **Extreme Programming에서 Iteration의 구성**
  - 소비자가 **높은 우선순위부터 'Use Case(활용 사례)'를 고른다.**
  - 그 사례들을 **Task들로 분할**한다.
  - 각 **Task에 대해 test case들을 개발**한다.
  - **Task를 구현하고, 전체 시스템의 설계를 진행**한다.
  
- **Extreme Programming의 Time Scale**
  - Release : 달 단위로 걸린다.
  - Iteration : 4주 이내로 걸린다.
  - Task : Day 단위
  - Test Case : Hour 단위
**~>** 즉, **빨리 걸린다!**

<br>

- **소비자와 프로그래머가 함께 계획**하는 특징이 있다.(Agile)

- **Integration 과정이 연속적**이다.

- **Simple한 디자인을 선호**한다.

- **테스팅 과정이 많다.**
  - **구현 전에 Unit Test**를 진행한다.
  - **JUnit**같은 것으로 **자동 테스트**를 한다.
  - **소비자가 직접 시스템**을 테스트해본다.
  
- ** XP**라고도 줄여부른다.

- _**Pair Programming, Collective Ownership, On-Site Customer, Open Workspace, Coding Standards(를 지킴)가 특징**_이다. 즉, 현대적 개발환경을 떠올리면 된다.

- **장점**
  - **리스크가 적다.**
  - **오버헤드가 적다.**
  - **변화되는 요구사항에 대응하기 쉽다.**
  - 시스템이 **간단**하다.
  
- **단점**
  - 소비자가 **함께 있어야한다.**
  - **크거나 복잡한 프로젝트에는 맞지 않는다.**
  - **자동적으로 테스트하기 어려운 프로그램의 개발엔 적합하지 않다.**
  
<br>

### (7) Hacking
&nbsp; SW 개발 시 **일련의 체계 없이 주먹구구식으로 코딩**만 하는 스타일을 의미한다.
**~>** 시스템 엔지니어링, 요구사항 분석, 설계, 검증 등의 과정 없이 코딩만 한다.
**~>** **구현에만 집중**한다.
