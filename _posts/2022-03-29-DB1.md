## Database System이란?
&nbsp; 컴퓨터공학에서 **'Database Management System', 약칭 'DBMS'**은 매우 중요하고 핵심적인 개념으로, **"서로 연관된 Valuable한 Data의 모음과, 그러한 데이터에 접근하는 일련의 프로그램들을 편리하고 효율적으로 다룰 수 있는 환경"**을 의미한다.

&nbsp; 데이터베이스 시스템에서 다루는 '데이터'는 다음과 같은 특성을 보인다.
- **Valuable : 가치 있는 데이터**를 다룬다.
- **Large : 큰 규모의 데이터**를 다룬다.
- **동시에 여러 사용자와 Application이 해당 데이터에 접근**한다.

**~>** Database System은 바로 이러한 '크고, 복잡한 데이터의 집합'을 관리하는 복잡한 SW 시스템을 의미한다. 

<br>

## Where?
&nbsp; 이러한 DBMS는 우리 삶에 매우 밀접하게 관련되어 있다. 셀 수 없을 정도로 많은 분야에서 DB를 사용하는데, 그 예를 몇 가지 들어보자면 다음과 같다.
- 기업에서의 DB : 소비자 정보, 제품 정보, 구매 정보, 세금/정산/자산 등의 회계 감사 정보, 직원/급여/감사 등에 대한 인적 자원 정보 등

- 제조업에서의 DB : 생산 라인, 생산 주문, 보급, 공급망 등에 대한 정보

- 은행 및 금융에서의 DB : 소비자 정보, 계좌, 대출, 거래 등의 정보, 신용 카드에 대한 정보, 주식, Real-Time Market 데이터 등

- 학업에서의 DB : 학생의 등록, 학점, 휴학, 수강 등의 정보, 교수 및 강사진에 대한 정보 등

- 항공에서의 DB : 예약 및 일정 등의 정보

- 통신에서의 DB : 전화 기록, 문자 기록, 명세서 등의 정보

- 웹 서비스에서의 DB : Amazon이나 쿠팡과 같은 인터넷 쇼핑몰 사이트에서의 각 종 정보 및 소비자 동향 등에 대한 정보, 광고 등

&nbsp; 등등 **수 없이 많은 분야에서 DB는 기능의 핵심으로서 존재**한다.

## DB의 목적
&nbsp; **20세기 중후반까지만 해도, 데이터베이스 시스템은 '파일화'**를 의미했다. 정보를 파일에 저장하고, 파일을 쌓아두는 방식으로 말이다. 이는 현대의 DBMS가 의미하는 바와는 확연히 다르다.

### **※ 초창기 Database의 단점**
**1) Data Redundancy(중복) : 중복된 데이터**가 여기 저기의 파일에 흩어져 있는 경우가 많았다.

**2) Data Inconsistency(비-일관성)** : 위와 같은 이유로, **동일한 정보가 서로 다른 파일에 저장되어 일관성이 유지되지 못할 가능성**이 농후했다.

**3) Access Difficulty(접근 비효율성)** : 데이터에 직접 접근해야했기 때문에, **새로운 데이터 접근이 필요할 경우 Access를 위한 새로운 프로그램을 매번 작성**해야하는 번거로움이 있었다. 즉, 비효율적이었다.

**4) Data Isolation(고립성)** : 말 그대로 데이터가 고립된 정도가 높다는 것이다. 이에 대한 개념은 추후 설명할 것이지만, 간단히 말하자면, **유사한 데이터가 복수의 파일에 통일되지 않은 형태로 있는데, 이들이 '유사하다'라고 취급되지 않고 각각 고립적으로 존재해 비효율성을 높이는 현상**을 말한다. 초창기 DB의 단점 중 하나였다.

**5) Integrity Problem(통합 문제)** : 각 데이터마다 접근 프로그램이 필요하기 때문에 '통합'과 관련한 문제가 있었다. **데이터 통합을 위한 제약 조건이 프로그램에 내장(Constraints are buried in program code)되어 있다보니 이를 명시적으로 확인하기 어려우며, 나아가 새로운 조건이나 기존의 조건을 변경하고자 할 때 작업이 번거로웠다. **

**6) NO Atomicity of Updates(업데이트의 원자성의 결여)** : **Atomicity, 원자성**은 추후 설명할 DBMS의 장점이다. **원자성을 좀 더 직관적으로 표현하면 '신뢰성'**, 혹은 **'멈추지 않는 정도'**라고 할 수 있는데, 초창기의 DB는 이러한 원자성이 결여된 특징이 있다. 
**~>** 예를 들어, 100만원을 A에서 B로 이체한다고 하면, 이체의 성공/실패와 상관없이 돈을 보내는 쪽은 성공했는데 가는 중간에 실패해 돈을 받는 쪽이 실패한다면, 이에 대한 정보가 불명확해진다. 이를 막는 것이 원자성이며, 초창기 DB는 이러한 능력이 부족했다.

**7) NO Concurrent Access(동시 접속의 어려움)** : 현대 DBMS의 가장 뚜렷한 장점은 바로 복수의 사용자가 데이터에 동시에 접근하는 것을 지원한다는 점이다. DB의 좋은 성능은 이 능력의 정도가 좌우한다고 해도 과언이 아닌데, 이러한 Concurrent Access가 보장되지 않는다면, 정보의 비-일관성이 높아지게 된다.

**8) Security Problem(보안 문제) : 사용자에게 서로 다른 권한을 주는 기능이 부족**했다. 특정 사용자에게 일부 데이터만 보여주고, 전체 데이터에는 접근하지 못하게 하는, 그러한 기능이 없었다는 것이다.

_**=>** 현대의 DBMS는 상기한 8가지의 모든 문제점에 대해 Solution을 제공한다. _

<br>

**※ 대학교 Database 예시** => 가장 피부에 와닿는 DB의 예시이다.
**~>** 대학 DB에는 학생, 강사, 수업에 대한 정보가 필요하다.
**~>** 그리고, 나아가 새로운 학생/강사/수업을 추가하는 기능이 필요하다.
**~>** 또한, 수강 신청과 수업 생성의 기능이 필요하다.
**~>** 학생에게 점수를 부여하고, GPA(Grade Point Average)를 계산해 성적표를 제공하는 기능도 필요하다. 등등

&nbsp; 앞으로 본 연재는 '대학 DB'를 기준으로 한 예제를 많이 사용할 예정이다.

## Data Models
&nbsp; **데이터베이스 시스템의 주요 목적은 사용자에게 Abstract View of the Data를 제공하는 것**이다. 이와 관련한 주요 개념이 두 가지 있는데, 그것은 바로 **'Data Model'과 'Data Abstracton'**이다.

- **Data Model** : **데이터와, 데이터 간의 관계, 데이터의 의미론(Semantics), 그리고 일관성 제약 조건(Consistency Constraint) 등을 설명하는 개념적인 도구의 모음**을 데이터 모델이라고 한다.

  - **Relational Model(관계형 모델)** : 우리가 다루는 **대부분의 DB**가 여기에 속한다. **술어 논리(Predicate Logic)와 집합론에 기반**을 둔 데이터베이스 모델이다.
  **~>** 모든 데이터가 Column과 Row의 **Table**에 저장된다. (Tabular Data)
  **~>** 1981년 Ted Codd가 제시하였다. 이전까진 후술할 네트워크나 계층형 모델이 주류였다.
  **~> Microsoft Excel**을 생각하면 된다.
  
  ![](https://images.velog.io/images/junttang/post/923b3c43-c0ba-4c02-a1bd-05ef8595c6ef/360px-Relational_model_concepts.png)
  
  
  - **Entity-Relationship Data Model(실재-관계 모델, ER 모델)** : Database를 **최초 Design할 때 데이터베이스의 논리적 구조를 ER모델로 설계하곤 한다.** 아래의 다이어그램을 보자. 익숙할 것이다.
  
  ![](https://images.velog.io/images/junttang/post/68450cec-7ae9-42b4-943e-432db8634767/450px-ER_Diagram_MMORPG.png)
  
  - **Object-Based/Oriented/Relational Data Model(객체지향형 모델)** : 추후 다룬다.
  
  - **Semi-Structured Data Model(반구조적 모델)** : **구조화가 완벽하게 되지 않은 데이터를 처리**할 때 사용한다. **XML**이 대표적인 예시이다.
  
  - **Others : Network Model, Hierarchical Model ~> 초창기 모델들. 사장되었다.**

## View of Data(Architecture of DB)

![](https://images.velog.io/images/junttang/post/9e3446b1-af71-4399-a4e4-72f9b0b62313/what-is-data-abstraction-in-dbms-and-what-are-its-three-levels-data-abstraction-levels-771e879679d76dea.jpg)

&nbsp; 위의 이미지는 **Database System의 일반적인 Architecture**를 보여준다. 그림처럼, **여러 개의 View가 있다. 하지만, 결국 이들은 하나의 Logical & Physical Level로 나타낸다. **

&nbsp; **Logical Level은 전체적인 데이터의 구조, 즉, 테이블의 구조**를 의미한다. 이러한 **로지컬 레벨 구조는 'File'의 형태로 Physical Level로** 들어가는데, _**사용자는 이 Logical & Physical Level에는 Access하지 않는다(DB의 특징).**_

<br>

## Instances & Schemas

- **Logical Schema** : **DB에 대한 전반적인 논리적 구조**를 의미한다. 일반적으로 로지컬 스키마를 _**프로그램의 '변수 Type'과 비유**_하곤 한다.

- **Physical Schema : DB에 대한 전반적인 물리적 구조**를 의미한다. _**File**_의 형태이다. 

- **Instance : DB의 Actual Content**를 의미한다. 특정 시간의 특정 위치에 있는 **실제 데이터**를 말한다. 일반적으로 인스턴스를 _**프로그램의 '변수의 값'**_과 비유하곤 한다.

**=>** 요약하자면, 로지컬 스키마는 자료형을, 인스턴스는 테이블에 저장된 실제 값을, 피지컬 스키마는 그 정보 테이블이 담기는 File을 의미한다고 보면 된다.

**ex)** 12개의 행(Row)이 있는 Table의 경우, 12개의 Instance가 저장된다고 볼 수 있다. 각 열은 그 Row에 대한 Detail한 Data들이다. Column들의 배열은 곧 Logical Schema를 보여준다고 볼 수 있다.

![](https://images.velog.io/images/junttang/post/275f2a1d-321a-412b-b894-ec5d83992c04/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

## Physical Data Independence
&nbsp; **로지컬 스키마를 바꾸지 않고 피지컬 스키마를 변형하는 기능**을 의미한다.

![](https://images.velog.io/images/junttang/post/1ee1a876-a9e7-4643-8ccf-a38469afa0a9/dbms-data-independence.png)

**~> 응용 프로그램은 Logical Schema에 의존해서 작성**된다. 따라서** Physical Schema를 바꾸더라도 응용 프로그램의 코드는 전혀 바꿀 필요가 없다. Physical Data Independence가 지켜지면 말이다.**

**~>** 만약 이 기능이 지원되지 않으면, 파일의 구조를 바꾸고 싶을 때 응용 프로그램의 코드 자체를 싹 바꿔야한다.

**=> 각 Level과 Components 사이의 인터페이스는 '잘' 정의되어야 한다. 각 부분에서 Change가 있더라도 다른 요소에 크게 영향을 주지 않게 하기 위함**이다. 효율적인 관리를 위해선 이러한 기능이 반드시 지원되어야 할 것이다.

<br>

## Data Definition Language(DDL)
&nbsp; **데이터베이스 스키마를 정의할 때 사용하는 특정 Notation, 언어를 의미**한다. 좀 더 좁게 말하자면, **객체의 생성/변경/삭제에 관한 명령어**를 의미한다. SQL에서 데이터 정의 언어의 문장은 관계형 **데이터베이스의 구조를 정의**한다.

```
create table instructor (
ID 			char(5),
name 		varchar(20),
dept	 	varchar(20),
salary 		numeric(8,2))
```

&nbsp; **DBMS에서는 DDL로 테이블 Template을 만들 수 있다.** 이러한 Table Template의 집합은 **'Data Dictionary'에 저장**된다.

&nbsp; 이때, **Data Dictionary는 Metadata를 가진다. Metadata란, 'Data about data'**로, 데이터에 대한 데이터를 말한다. 위의 예시에서 **varchar이나, char, 이런 요소들** 말이다.

<br>

**※ Metadata** : 엄밀히 말하면, **데이터에 관한 구조화된 데이터**로, 다른 데이터를 설명해 주는 데이터이다. **대량의 정보 가운데에서 찾고 있는 정보를 효율적으로 찾아내서 이용하기 위해 일정한 규칙에 따라 콘텐츠에 대하여 부여되는 데이터**이다. 이러한 메타 데이터에는 다음과 같은 요소들이 포함된다.
 - **Database Schema**
 - **Integrity Constraint** **ex) Primary Key (Column)** : 각 Instance를 **"유니크하게" 구분**할 수 있게 하는 기준을 의미한다. 위의 Instructor 구조에서는 ID가 바로 이러한 Primary Key 역할을 할 수 있다.
 - **Authorization : 데이터에 대한 접근 권한 정보**를 의미한다.
 
 <br>
 
_**=> DDL로 정의한 데이터베이스 스키마는 메타 데이터이며, 데이터 사전에 포함되고, 그러한 데이터 사전에는 스키마 외에도 프라이머리 키, 권한 설정 등의 메타 데이터가 있다는 것**_이다.

<br>

## Data Manipulation Language (DML)
&nbsp; 적절한 **데이터 모델에 의해 구조화된 데이터를 접근하고 업데이트하는 데에 쓰이는 언어를 의미**한다. 데이터베이스 사용자 또는 응용 프로그램 SW가 컴퓨터 데이터베이스에 대해 **데이터 검색, 등록, 삭제, 갱신**을 위해 사용하는 데이터베이스 언어 또는 데이터베이스 언어 요소를 말한다. 

&nbsp; DML은 아래와 같이 두 종류로 구분할 수 있다.

- **Procedural DML(절차형 DML)** : **사용자가 'What data are needed'와 'How to get those data'를 정의**해주어야 하는 언어이다. DBMS에서는 구형의 언어라고 할 수 있다. 프로그래밍 언어의 정의 구분과 유사하게 보면 된다.

- **Declarative DML(선언형 DML, 비-절차형 DML)** : **'How to get those data' 없이 사용자가 'What data are needed'만 명시**해도 되는 언어이다. 즉, 내가 필요한 데이터가 무엇인지만 작성하면 된다는 것이다. 접근을 위한 절차, Step은 작성할 필요가 없는 언어이다.
**~> Declarative DML은 Relational Model이 나오고 나서 가능해졌고, 그것에 적합**하다.

_**=> Procedural = "What I want" + "How",   Declarative = "What I want"**_

<br>

&nbsp; 프로그래밍 언어와 마찬가지로, 선언형 DML이 절차형 DML보다 확실히 사용자 입장에서 편리하다. 

&nbsp; _**'정보 검색' 기능이 있는 DML을 'Query Language'라고 부른다.**_

<br>

## SQL Query Language
&nbsp; **SQL Query Language는 비-절차형(선언형) 언어**이다.
&nbsp; &nbsp; SQL 쿼리는 **여러 개(혹은 단일)의 테이블을 입력으로 받아 항상 단 하나의 결과 테이블을 출력**한다.

```
select name				// 모든 컴퓨터공학과 강사를 찾는 명령
from instructor
where dept_name = 'ComputerScience'
```

&nbsp; **SQL은 튜링 머신과 조응하는 언어가 아니다. Imperative & Procedural이 아니므로.**

_**~> SQL은 그 자체로서는 간단한 선언형 언어이기 때문에 복잡한 함수를 구현하기 위해선 일반적으로 HLL에 혼합되어 쓰인다.**_

**~> 그래서, DB 응용 프로그램은 일반적으로 '임베디드 SQL을 지원하는 HLL'와 'DB에 쿼리를 전달하는 응용 프로그램 인터페이스 ex. ODBC/JDBC'를 통해 DB에 접근**한다.

<br>

## Application Program으로 DB 접근하기
&nbsp; **SQL과 같은 비-절차형 쿼리 언어은 아무래도 Declarative이므로 일반적인 Turing machine에 비해 기능이 제한적**이다. 예를 들면 다음과 같은 요소들이 있는데,

- SQL은 **사용자로부터 입력을 받거나, Display에 출력을 하거나, 네트워크를 통해 통신하는 등의 기능을 제공하지 않는다.**

- 이러한 SQL **비-지원 기능들은 C/C++이나 Java, Python과 같은 'Host Language'로 작성되어야 한다.** _'호스트 언어'라 함은, Embedded SQL을 지원해 DB에 대한 접근을 가능케하는 기반 언어_들을 의미한다.

- Application Program : 이러한 맥락에서, 응용프로그램이란, DB와 소통할 수 있는 프로그램들을 통칭한다.

<br>

## DB 설계
&nbsp; 데이터베이스의 구조를 설계하는 과정에 대해 간단하게 정의해보자.

- **Logical Design** : **데이터베이스 스키마를 결정하는 과정**을 의미한다. 우리가 '좋은 관계 스키마의 모음(Good collection of Relation Schemas)'을 찾는 과정을 필두로 한다.
  - **Business Decision : DB에 어떤 요소들을 기록할지**를 결정한다. 쉽게 말해 _**"어떤 Column이 필요할까?"**_를 결정하는 것이라 보면 된다.
  - **Computer Science Decision : 어떤 Relation Schema들을 택할지, 그리고 그들로부터 어떤 요소를 뽑아내 분류할지 등을 결정**하는 과정이다. 쉽게 말해 "_**어떤 테이블을 몇 개 구성할지**_, 그리고 이들을 어떻게 구분할지, _**특정 테이블에는 어떤 Column들을 포함시킬지**_" 등을 선정하는 과정이라 보면 된다.
  
- **Physical Design : DB의 물리적인 레이아웃을 결정**하는 과정이다.

<br>

## DB 엔진
&nbsp; **DB System은 전체 시스템의 각 Responsibility들을 처리하는 모듈들로 분할**할 수 있다. 이러한 DB 시스템의 '기능적 요소'들은 주로 다음과 같이 분류한다.

- **The Storage Manager (저장)**
- **The Query Processor Component (검색)**
- **The Transaction Management Component (상호작용)**

<br>

### (1) Storage Manager
&nbsp; **DB에 저장된 Low-Level Data와 응용프로그램, 시스템에 도착한 쿼리들 사이의 인터페이스를 제공하는 프로그램 모듈**을 'Storage Manager'라고 한다.

&nbsp; 스토리지 매니저는 다음과 같은 작업들을 수행해야한다.
- **OS File Manager와의 소통**
- **효율적인 '데이터의 저장, 삭제, 업데이트'**

<br>

&nbsp; 스토리지 매니저는 다음과 같은 요소들로 이루어진다.
- **Authorization(권한 설정)과 Integrity(통합) Manager ~> 주의!**
- **Transaction Manager**
- **File Manager**
- **Buffer Manager** : 디스크에서 데이터를 읽어올 때, 메모리의 버퍼에 있는 것을 미리 가져와서 읽고자 이러한 기능을 담당함.

<br>

&nbsp; 스토리지 매니저는 Physical 시스템 구현에서 몇 가지 _**자료구조**_를 구현한다.
- **Data Files : DB 그 자체를 저장**한다.
- **Data Dictionary : DB 구조에 관한 메타 데이터들을 저장**한다. 특히나, 데이터베이스 스키마를 저장한다.
- **Indices : 데이터 아이템들에 대한 빠른 접근을 가능케 하기 위해 '인덱싱'을 제공**한다. 특정 값을 지닌 아이템들을 가리키는 포인터들을 인덱스로 갖추고 있다.

<br>

### (2) Query Processor Component
&nbsp; 쿼리 처리 요소는 다음과 같은 요소들을 포함한다.
- **DDL 인터프리터** : **DDL 문장들을 인터프리팅하고, Data Dictionary에 해당 DB스키마를 저장**한다.
- **DML 컴파일러** : 쿼리 언어로 된 **DML 문장들을 'Evaluation Plan'으로 변환**한다. (이름이 컴파일러일 뿐, 하는 역할은 인터프리터의 그것과 같다.)
  - **Evaluation Plan : Query Evaluation Engine이 처리하는 Low-Level 명령들**
  - **DML 컴파일러는 'Query Optimization'을 수행**한다. 말 그대로 '최적화'인데, 정확히 무엇이냐면, 변환 가능한 **Evaluation Plan들 중 가장 Cost가 낮은 명령을 택해 변환**한다는 것이다.
  
- **Query Evaluation Engine** : DML 컴파일러로 **변환된 Low-Level 명령들을 수행**한다.

<br>

### **※ Query Processing 과정**
**1) Parsing and Translation** : 입력받은 쿼리를 **Parser**가 분석하여 SQL 파싱 트리를 만들고, **Translator**가 문법 및 의미 오류 체킹 후 **Relational-Algebra 표현식**으로 변환한다.
**2) Optimization** : **Optimizer**가 DB의 데이터 정보를 바탕으로 Cost를 계산하면서 **Query 실행 계획(Plan)**을 세운다. 이때, **'Statistics about data'란, Metadata**라고 보면 된다.
**3) Evaluation** : **Evaluation Engine이 Plan을 바탕으로 Query를 실행하여 결과를 반환**한다.

![](https://images.velog.io/images/junttang/post/ef0892bf-da71-4514-bb52-d894e7e4706e/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

<br>

### (3) Transaction Management Component
&nbsp; **데이터 응용 분야에서 단일 Logical Function 역할을 수행하는 Operation들을 Transaction이라고 한다. 이를 처리하는 Manager**를 일컫는다. 이는 (1)인 Storage Manager에 포함된 요소이다.

- **Transaction Management Component는 전원이나 OS충돌 등으로 발생하는 System Failure나, 또는 Transaction 자체로 인해 발생하는 Transaction Failure가 발생하여도 DB의 상태를 일관성 있게, 정확하게 유지하는 기능을 보장해야한다.**

- **Concurrency-Control Manager**가 있는데, 이는 **Concurrent Transaction들의 상호작용을 관리**한다. **DB의 Consistency를 유지하기 위해서 이러한 기능을 수행**한다.

**~>** Transaction Management Component에 대한 자세한 설명은 추후 하도록 하겠다. 일단 그 존재 정도만 기억하자.

<br>

## DB Architecture
&nbsp; DB를 설계할 때, **DB의 CS적 구조를 어떻게 구성했느냐에 따라 아래와 같이 분류**할 수 있다.

- **Centralized DB : 적은 CPU(Core)와 공유된 메모리를 특징**으로 한다. 적은 CPU에 집중시켰다 해서 Centralized라 하는 것이다.

![](https://images.velog.io/images/junttang/post/f708ab74-0d7c-48fe-a956-999aacebd94c/Capturecentraldb.png)

![](https://images.velog.io/images/junttang/post/5c00f5b0-4619-45e9-b567-7288db286ac1/asdsadasd.png)

(Centralized DB의 구조 그림이다. DDL 인터프리터, DML 컴파일러, Evaluation Engine 등으로 구성된 쿼리 프로세서와, 버퍼/파일/Transaction 매니저와 권한 및 통합 매니저로 구성된 스토리지 매니저가 있고, 스토리지 매니저가 구현한 데이터 사전과 메타 데이터, 인덱스, DB 파일 등이 Disk Storage에 있는 형태에 주목하자.)

<br>

- **Client-Server 구조 : 복수의 Client 머신들에서 요청하는 일들을 하나의 서버 머신이 처리하는 형태의 DB 구조**를 의미한다.

![](https://images.velog.io/images/junttang/post/eeb1883b-99b1-472c-9e34-b84c8b79d4d0/asdasdasdasd.png)

- **Parallel DB : CPU가 여러개인 DB 구조**라 볼 수 있다. 
  - 다수의 CPU(Core)와 공유된 메모리를 특징으로 한다. (**Shared Memory with Many cores**)
  - 공유된 디스크를 특징으로 한다. (**Shared Disk**)
  - 이때, **디스크와 CPU는 서로 별도로 존재한다. 이를 'Shared Nothing'**이라 한다.
  
![](https://images.velog.io/images/junttang/post/4e3d8c5c-93c4-4b65-93fb-d391ef833721/asdasdasdasdasd.png)
  
- **분산(Distributed) DB : DB가 여러 지역으로 분포해있으면서 통합관리되고 있는 형태의 DB**를 일컫는다. 아래의 특징들을 가진다.
  - **지리적 분산(Geographical Distribution)** : 여러 지역에 나뉘어 DB가 존재한다.
  - **스키마/데이터 이질성(Schema/Data Heterogeneity) : 스키마와 데이터가 지역마다 이질적인 특징**을 보인다.
  **ex)** 서울 DB엔 서울 고객들, 부산 DB엔 부산 고객들이 있는데, 두 지역의 고객 성향이 달라서 스키마와 데이터에 차이가 있다는 것!!
  
![](https://images.velog.io/images/junttang/post/f001136c-1b43-4d53-9e73-4c91c9f7c2fa/vd.png)

## DB 응용프로그램 구조
&nbsp; DB 응용(Application)은 일반적으로 아래의 두 종류로 분류한다.

- **2-Tier Architecture : 응용프로그램이 Client 머신에 존재하고, 이것이 Server 머신에 있는 DB 시스템을 끌어와 동작**시키는 형태이다.

- **3-Tier Architecture** : Client 머신이 직접 Server 머신의 DB와 소통하지 않고, **중간에 'Application Server(Interface라고 보면 된다)'를 두어 DB와 소통하는 형태**를 말한다. 이 형태가 _**투-티어에 비해 더 Light하다.**_

## DB 사용자 유형
&nbsp; DB 사용자는 다음과 같이 분류할 수 있다.
- Naive User : DB의 고객을 의미한다. ~> 응용프로그램, 인터페이스를 사용
- Application Programmer : DB의 응용프로그램을 만드는 개발자 ~> 응용 프로그램 개발
- Sophisticated User : DB 분석가와 같은 고급 사용자를 의미한다. ~> 쿼리를 사용
- DB Administrator : DB에서 주요 권한을 가진 사람을 의미한다. ~> 권한 툴과 쿼리를 사용

**※ DB Administrator : DB 시스템 전반에 대한 처리 권한을 가진 사람**을 의미한다. 주로 약칭인 **'DBA'**로 많이 불리는데, 이러한 DBA가 하는 일은 다음과 같다.
```
- 스키마 정의
- Storage 구조와 접근방법 정의
- 스키마와 물리적 구조 수정
- 데이터 접근 권한 허가 및 조정
- 유지보수
- 주기적인 백업
- 디스크 공간이 충분한지 확인 및 조정
- DB 전반을 모니터링하고 관리
```
