## Programming Language

### 1. 왜 PL 개념을 배울까?

**1. 'Idea'를 표현하는 능력을 키울 수 있다.**

&nbsp; 특정 기능이 결여된 언어를 사용해야하는 상황에 놓여있다고 상상해보자. 나는 현재 그 기능이 필요한데, 그 기능을 가진 다른 언어를 학습하면 현재 내가 사용해야하는 언어를 배우는 과정에서 좀 더 도움이 될지 안될지는 사실 알 수 없는 문제이다.

&nbsp; 하지만, 내가 사용해야하는 언어가 '그 기능'을 Subprogram같은 형태로 제공하지 않는 것일 뿐, 어떻게든 구현은 할 수 있다. 왜냐? 프로그래밍 언어이기 때문이다. 따라서, '그 기능'을 가진 다른 언어를 공부해보면 좀 더 도움을 얻을 확률이 높다고는 말할 수 있다.

&nbsp; 우리는, 바로 이러한 점에서, **PL 개념을 잘 익혀두면 다양한 언어를 능숙하게 다루는데 많은 도움을 얻을 수 있다. 그리고 이는 곧 'Idea'의 표현 능력과 직결**되는 능력일 것이다.

<br>

**2. '적합한(Appropriate)' 언어를 고를 수 있는 능력, 경험을 갖게 된다.**

&nbsp; **프로그래머가 몇 가지 언어에 능통하고, 그 언어들의 용도와 목적, 특징에 대해서도 잘 알고 있다면, 그 프로그래머는 다양한 문제 상황에서 적합한 언어를 고르는데 보다 더 유리한 위치를 선점**할 수 있다.

&nbsp; 우리는 C언어로도 어떻게든 AI, 과학계산용, 비즈니스용 프로그램을 만들 수 있다. 하지만, "특정 용도에 더 적합한" 언어란 것은 반드시 존재한다. 컴파일 방식과 내장 함수 등의 여러 이유로 '더 적합한'이라는 개념이 실현된다.

&nbsp; **ex)** Pascal은 Matrix 계산을 단순한 사칙연산 기호로 처리할 수 있고, 소스 코드가 컴파일 될 때 CPU 구동 원리에 맞게 적절하고 효율적으로 기계어 명령 변환을 수행한다.
**~>** 물론 C언어로도 충분히 행렬 연산을 구현할 수 있지만, 직접 스스로 알고리즘을 설계해야하고, CPU 구동 원리를 고려해 효율적으로 돌아가게 짜는 것은 더더욱 어려운 일이다.

<br>

**3. 새 언어를 습득하는 능력이 좋아진다.**

&nbsp; **프로그래밍 언어들 전반에서 나타나는 Fundamental Concepts들을 깊게 이해**하고 있다면, **새 언어에 들어있는 기능, 그 기능이 구현된 원리 등을 깨우치기가 훨씬 쉬워진다.**

<br>

**4. '구현의 중요성(Significance of Implementation)'을 더 잘 이해할 수 있다.**

&nbsp; '구현의 중요성', '구현 능력 향상'이라 함은, 함수의 사용 능력, **'Programming Methodology', 'Modular Programming'**과 직결된다고 할 수 있다. 모듈화 프로그래밍 능력이 높아지면 우리는 **언어를 더 효율적으로 사용**할 수 있다.

**~> 물론, 함수화, 모듈화한다고 해서 꼭 반드시 효율적이라고 할 순 없다.** 사소한 코드 블록까지 모듈화하면, 괜히 _**함수 호출/반환이 잦아져 오히려 더 늦어질 수 있다.**_ 함수 호출/반환에 쓰이는 어셈블리명령만 해도 약 10개 이상이다.

**~>** 단순히 'Modular Programming'만이 '구현 능력'은 아니다. 다음과 같이 '인덱싱 이해'도 구현 능력 중 하나이다. **예를 들어, a[i][j]와 같은 배열 요소를 접근한다고 할 때, 만약 이 요소가 저장된 메모리가 Main Memory가 아니라 Disk(or SSD)인 경우, Page Fault로 인한 접근 시간 저하**가 발생한다. 따라서 과도한 중복 인덱싱은 효율면에서 바람직하지 않은데, 이러한 사실들을 이해하는 것 역시 '구현 능력 향상'의 예시라 할 수 있다.

### 2. 프로그래밍 언어의 용도별 구분(Programming Domains)

1. **Scientific Applications**
- 과학연구용 프로그래밍 언어는 **자료구조는 간단하지만, 대량의 실수 연산이 필요**하다는 특징을 보인다.
- 연산의 효율성이 중요한데, **FORTRAN**이 아주 대표적인 과학용 프로그래밍 언어이다.

<br>

2. **Business Applications**
- 비즈니스용이다 보니 아무래도 **정교한 입출력 기능과 다양한 10진수 자료형**들의 유무 등이 중요하다.
- **COBOL**이 대표적이다. 코볼 외에는 흔치 않다.

<br>

3. **Artificial Intelligence Applications**
- AI 인공지능에 적합한 프로그래밍 언어이다. **Symbolic Processing과 List 자료구조 등이 특징**이다.
- **LISP와 PROLOG**가 대표적인 인공지능용 개발 언어이다.

<br>

4. **System Programming**
- 시스템SW는 **끊임없이 연산을 수행하므로 '연산 효율성(Execution Efficiency)'이 중요**하다.
- **Low-Level한 특성**을 보인다.
- **'주변 장치 드라이버(Peripheral Device Driver) SW'를 작성**하는데 쓰일 수 있다.
- C언어가 대표적이다.
  - **C언어로 개발된 UNIX의 경우, Easy to port, Easy to move가 특징**으로, 높은 호환성을 자랑한다.
  
<br>

5. **Web Software**
- **XHTML + Scripting Language ex) shell, ask, tcl/tk, perl, JavaScript** 등
**~> Dynamic Contents**들을 표현하는것이 주된 기능이다.

- **Web Brouser 안에 JavaScript 인터프린터가 내장**되어 있기 때문에 자바스크립트로 만든 홈페이지가 Web Brouser 상에서 구동되는 것이다. 이런 것을 WebOS라고 하는데, _**Flatform Independent**_한 것이 특징이다(**웹브라우저만 있으면 어떤 장치에서든 홈페이지 동작 가능**).
- **주로 인터프린터 방식**으로 구성된 언어가 많다. Runtime때 수행하는 방식으로.

<br>

**여담)** Client-Side Programming, Web 개발이 Front-End Developer의 특징이고, MySQL, Oracle 따위로 DB 구성, Sever 유지 등을 하는 것이 Back-End Developer의 특징이다.

<br>

**여담)** 참고로, 어셈블리어로 위에 상기한 모든 목적의 프로그램들을 만들 수 있다. (고급) 프로그래밍 언어들이 등장한 이유는, "좀 더 쉽게 ~용의 프로그램을 개발할 수 있는 언어를 만들어보자"라는 아이디어가 바탕이 된 것이다. 왜냐? 어셈블리어로 만드는 과정은 너무 Naive하므로.

<br>

**여담)** 어떤 개발자가 **만약 어셈블리언어에 능통하다면, 사실 상기한 HLL보다 어셈블리어로 짜는 것이 훨씬 더 빠른 프로그램을 만들 수 있는 방법**이다. 왜냐하면, HLL의 경우, 컴파일이나 인터프리팅 과정에서의 불필요한, 또는 소모적인 변환이 있을 수 밖에 없는데, 인간이 이를 제거해 더 빠른 프로그램을 만들 수 있다.
**~> 하지만, 현대 프로그래밍에서는 Readability와 Writability가 매우 중요**하고, HW의 성능이 워낙 좋아졌기 때문에 굳이 이럴 필요는 사실상 전무하다.
**~>** 조금 가까운 옛날로 돌아가보면, 과거 많은 인기를 누렸던 타이쿤 게임 '**Rollercoaster Tycoon 2**'가 바로 대표적인 어셈블리어 기반 프로그램이다. 아직까지도 이 게임이 높은 최적화 능력으로 큰 사랑을 받고 있는 것은 '어셈블리어의 특징'을 알면 이해할 수 있다.

### 3. 프로그래밍 언어 평가 기준
&nbsp; **프로그래밍 언어를 평가하는 다양한 지표가 있는데, 이러한 지표들을 통해 크게 4가지 측면의 평가 기준을 도출**할 수 있다.

<b>

```
Simplicity   				-> Readability, Writability
Orthogonality   			-> Readability, Writability
Control Statement   		-> Readability
Data Structure   			-> Readability
Syntax Design   			-> Readability
Support for Abstraction		-> Writability
Expressivity   				-> Writability
Type Checking   			-> Reliabilitiy
Exception Handling   		-> Reliabilitiy
Restricted Aliasing   		-> Reliabilitiy
Programmer Training   		-> Cost
Program Writing   			-> Cost
Compilation   				-> Cost
Execution   				-> Cost
Maintenance   				-> Cost
```
  
  </b>
