## 프로그래밍 언어 평가 기준
  프로그래밍 언어를 평가하는 다양한 지표가 있는데, 이러한 지표들을 통해 크게 4가지 측면의 평가 기준을 도출할 수 있다.
  
![](https://images.velog.io/images/junttang/post/7229d7cd-4ffc-445c-8095-b63e544d77f4/asdsad.png)

**Simplicity   				-> Readability, Writability
Orthogonality   			-> Readability, Writability
Control Statement   		-> Readability
Data Structure   			-> Readability
Syntax Design   			-> Readability
Support for Abstraction		-> Writability
Expressivity   				-> Writability
Type Checking   			-> Reliabilitiy
Exception Handling   		-> Reliabilitiy
Restricted Aliasing   		-> Reliabilitiy
Programmer Training   		-> Cost
Program Writing   			-> Cost
Compilation   				-> Cost
Execution   				-> Cost
Maintenance   				-> Cost**

- 이때, **Writabilbity와 Readability는 Reliability에 영향을 준다.**

**ex**) C언어에서 Division by Zero Error를 감지하는 기능인 Exception Handling은 대표적인 Reliability를 높여주는 특성이다. 실수를 줄여주기 때문이다.

**ex) 아무래도 Machine Language나, 그것에 가까울수록 Readability와 Writability가 떨어진다. 하지만 반대로 Cost 측면에서는 기계어가 더 우수**하다. 즉, _**Tradeoff**_가 있다는 것을 주목하자.

<br>

### 1. Readability (가독성)
&nbsp; **'소프트웨어 생명주기(Software Lifecycle)' 개념이 도입된 이후, 가독성(Readability)은 프로그램과 프로그래밍 언어를 평가하는 매우 중요한 기준** 중 하나로 떠올랐다.

> _**가독성이 중요한 이유 : 가독성은 프로그램의 Maintenance에 직결**되는 문제이기 때문이다._

&nbsp; 이러한 '가독성'은 다음의 다섯 가지 요소로 결정된다.
**=>** **Overall Simplicity, Orthogonality, Control Statements, Data Types & Structures, Syntax Consideration**

<br>

**(1) Overall Simplicity (전반적 편리함)**
&nbsp; &nbsp; 프로그래밍 **언어의 전반적인 편리함 그 자체**를 의미한다. 

- 프로그래밍 언어의 **문법에서 기본 요소들이 다양할수록 오히려 Simplicity에는 악영향을 준다.**
**ex) 똑같은 일을 하는데 여러가지 방법이 존재하면, 같은 기능을 두고 다양한 Variation이 발생**해 Simplicity가 떨어지고, 이는 **가독성의 저하**를 불러일으킨다.

  - **Feature Multiplicity : 하나의 특정 기능을 수행하는데에 여러 가지 방법이 존재**하는 것을 일컫는 말이다.
 ** ex) cnt++, ++cnt, cnt+=1** 등은 모두 같은 기능을 수행하는 서로 다른 표현법이다.
  
- **Operator Overloading은 Simplicity를 키워주는 기능**이다. 만약, 정수 더하기와 실수 더하기가 다른 Notation의 Operator를 사용한다고 해보자. 얼마나 복잡하겠는가?

- 하지만, 프로그래밍 **언어가 너무 과하게 Simple하면, 그것은 또 그 나름대로 단점**이 될 수 있다. **어셈블리어가 바로 대표적인 예시**로, 어셈블리어는 Feature Multiplicity가 별로 없다. 그럼에도 불구하고 아무도 어셈블리어가 편리하다고 말하지 않는다.

_=> 언어의 전반적 편리함은 가독성을 높여주는 요소이다._

<br>

**(2) Orthogonality (직교성)**
&nbsp; &nbsp; 프로그래밍 언어가 '직교적(Orthogonal)'이라는 말은 무슨 말일까?

> **여러 가지 기본 문법 요소가 서로 기능적인 면에서 겹치지 않고, 그 요소(기능)들을 조합했을 때 예외가 발생하지 않을 때, 그러한 속성을 Orthogonal하다고 한다.**

&nbsp; 즉, 중복되지 않는 기능들이 있고, 걔네를 조합했을 때 예외가 없으면 직교적인 프로그래밍 언어인 것이다.

> _Formal Definition : Orthogonality means that there is a "relatively small set of primitive constructs" that can be combined in a "relatively small number of ways" to build the control and data structures of the language._

**~> "Relatively Small", 즉, '상대적으로 적은' 수의 기초 문법만으로 '상대적으로 적은' 방식으로 프로그램 제어와 자료구조 설계를 문제없이 할 수 있을 때, 이를 '직교적'이라 한다.**

> _프로그래밍 언어가 직교적일수록, 예외가 적어진다._

**ex) C언어의 return()은 데이터 Type과 서로 직교적**이다. Type과 return은 서로 겹치지 않는 문법 요소이다. 그리고, int, char 등을 return과 조합했을 때 예외가 없다.

**ex) 하지만, return과 type이 '완벽히 직교적'인 것은 아니다.** 예를 들어, type이 array인 경우를 생각해보자. 배열의 시작주소값을 넘기는 방식 말고, 배열 그 자체를 리턴할 수 있는가? 그렇지 않다. 즉, 예외가 있는 것이다. 
**~> return과 type은 Orthogonal하지만, 완벽하게 Orthogonal한 것은 아니다.**

_=> 언어가 직교적일 때, 복잡한 표현이 불필요해지므로, 가독성이 올라간다._

<br>

**(3) Control Statements (흐름 제어문)**
&nbsp; &nbsp; **흐름 제어문 중 Jump하는 흐름 제어문이 없을수록 Readability가 올라간다.** 예를 들어, 어셈블리어의 Goto 흐름같은 것 말이다.

> _**Goto-less한 프로그래밍 언어일수록 Control Statements가 Readability를 높여준다.**_

- 프로그램의 흐름이 **Top에서 Bottom으로 흐르는 것이 무작위로 Jump하는 흐름보다 훨씬 더 가독성이 뛰어나다.** (당연한 서술)

&nbsp; 컴퓨터공학과에서 C언어를 처음 접하는 이들에게 "Goto문 문법을 알려주면서도, 이를 왠만해선 쓰지말라"라고 가르치는 이유가 바로 여기에서 기인한다. 정말 불가피한 상황, 예를 들어서 서로 다른 함수가 중첩적으로 호출되면서 특정 상황에 탈출해야하는 경우, 이러한 특수한 케이스를 제외하고는 goto문을 사용하지 않도록 한다. 가독성을 심하게 해치기 때문이다.

_=> 프로그래밍 언어의 흐름제어문에 따라 프로그램 코드의 가독성이 달라진다._

<br>

**(4) Data Types & Structures (자료형과 자료구조)**
&nbsp; &nbsp; **용도에 알맞는 자료형과 자료구조를 지원하는 기능은 가독성을 높이는데 일조**한다. 

&nbsp; 당연한 것이, C++에서 **stl을 사용하지 않고서 Linked List를 작성**해야한다고 해보자. 일일이 구조체를 만들고, 필요한 연산 모듈을 만들어야할 것이다. 물론, 이들이 좋지 않은 것은 아니지만, 적어도 '가독성'이라는 측면에서는, 애초에 이를 지원하는 언어가 더더욱 보기 편리함은 부정할 수 없다.

&nbsp; **int형, boolean형, enum 자료형 등등 모두 이러한 예시**에 해당한다.

_=> 적절한 자료형의 제공은 가독성을 높이는데 일조한다._

<br>

**(5) Syntax Consideration (구문 작성 시의 센스)**
&nbsp; &nbsp; 프로그래밍에서 매우 중요한 자질 중 하나는 **"변수와 함수 이름을 적절하게 작성할 수 있느냐?"이다. 이는 가독성에 직결되는 요소**이기 때문이다. 이를 'Syntax Consideration 능력'이라고 하자.

> _**프로그래밍 언어를 사용할 때, 변수나 함수 이름을 큰 제한 없이 자유롭게 쓸 수 있어야 소스 코드의 가독성을 높일 수 있다.**_

- **Indentifier(식별자)의 형태** : 변수명, 함수명 작성 방법을 결정
- **Keyword의 형태** : keyword가 직관적이어야한다.
- **생긴 것은 비슷한데, 그 기능이 다른 문법의 제공은 '가독성' 측면에서 좋은 프로그래밍 언어가 아니다.** 이유는? 간단하다. 헷갈려서이다. 대표적인 예시가 FORTRAN의 GO TO (,,,)

<br>

### 2. Writability (작성능력)
&nbsp; 국문으로 가장 가까운 의미를 떠올려 보면 '작성 능력'이겠지만, 이는 충분치 않다. 그냥 Writability라는 표현이 가장 잘 어울린다. "쉽고 다양하게 작성할 수 있게 인터페이스를 제공하는가?"라고 생각하면 좋을 듯 하다.

> _**Writability : "How easily a language can be used to create programs for a chosen problem."**_

&nbsp; 이러한 'Writability'는 다음의 네 가지 요소로 결정된다.
**=> Overall Simplicity, Orthogonality, Support for Abstraction, Expressivity**

<br>

**(1) Simplicity**
&nbsp; &nbsp; 앞서, 언어의 **Simplicity는 Low Feature Multiplicity와 Operator Overloading을 필두로 한다**고 했다. 이들로 인해 제고되는 '심플함'은 **당연히 코드 작성을 위해서도 좋은 장점**이 될 것이다. 쉽게 쓸 수 있기 때문이다.

**(2) Orthogonality**
&nbsp; &nbsp; 직교성 역시 마찬가지로 Writability를 높여준다. 겹치지 않는 기능들이 많고, 이들을 조합했을 때 예외가 없을 때, 그 언어는 직교적이라 했다. 언어가 직교적이면 마찬가지로, 당연히 코드 작성이 쉬울 것이다.

> _**Simplicity와 Orthogonality는 Readability뿐만 아니라 Writability의 향상에 도움이 된다.**_

**ex) 예를 들어 단순히 Primitive만 많고 이들이 직교적이지 않은 언어**가 있다고 해보자. **뭐 하나 구현할 때마다 예외가 빈번히 발생**하고 그러면, 이들을 신경써야하고,.. 매우매우 작성하기 불편할 것이다.

<br>

**(3) Support for Abstraction (추상화 기능 지원)**
&nbsp; &nbsp; 가독성을 높여주는 'Proper Data Types & Structures'와 유사한(같지는 않다. 바로 아래 서브루틴 이야기가 바로 그 이유) 이야기이다.

> _**추상자료형에 대한 지원 기능은 Writability를 높여주는 중요한 기능이다.**_

- 이때, **Abstraction은 단순히 자료구조 뿐만 아니라 함수도 의미**한다.
  - **Set of Codes를 추상화해놓는 것이 곧 Subroutine(함수)을 만드는 것**이고, 이러한 기능이 바로 **'Support for Abstraction'이다.** 함수 작성 기능이 있으면 당연히 작성 능력에 도움이 될 것이다.
  
- 각 기능에 대한 다양한 함수를 지원하는 언어라면 말할 필요도 없이 작성 능력에 도움이 될 것이다. **Code의 Complexity가 낮아질 것**이기 때문이다.

- **데이터 타입, 자료구조에 대한 추상화도 당연히 여기에 포함**된다.

**1) Process Abstraction : Subprogram, 즉, 프로시저, 함수 제공**을 이야기하는 것.
**2) Data Abstraction : Record Type, 즉, Class나, 자료구조의 제공**을 이야기하는 것.

**=> 이들은 모두 각각 함수와 자료형의 내부 Detail을 몰라도 잘 사용할 수 있게 도와준다. 그리고 당연히 이들은 Writability의 향상에 직결**된다.

<br>
<br>


**(4) Expressivity (표현 능력)**
&nbsp; &nbsp; 이는 **가독성을 떨어뜨리는 'Feature Multiplicity'에 관한 내용**이라 할 수 있다. 하나의 기능에 대해서 _**여러 가지 표현 방법이 있으면 가독성은 떨어질 수 있어도 Writability는 높일 수 있다.**_

**ex) 앞서 보여준 cnt++, ++cnt, cnt+=1** 예시
**ex) C언어의 for, while, do~while문** : 모두 결국 하나로 같은 기능을 제공할 수 있지만, 세부적으로 약간 다르게 조정하여 때에 따라 적절히 쓸 수 있게 하였음. 이는 표현 능력을 높이는 요소이자, 나아가 작성 능력을 높이는 요소임.

<br>

### 3. Reliability (신뢰성)
&nbsp; **프로그램이 모든 조건, 입력에 대해서 정상적으로 수행이 가능할 때, 우리는 이를 'Reliable'하다고 한다**.

- **어셈블리어가 바로 이러한 '신뢰성'이 떨어지는 언어**이다. 어셈블리어는 코드 레벨에서 Language Rule의 위반 사항을 확인 자체를 하지 않는다. 즉, 직접 우리가 찾아내야한다.

- 반대로, **에러 체킹을 하는 HLL의 경우 '신뢰성'이 높다.**

> _We say "A program is 'reliable' " when it performs its specifications under all conditions!_

&nbsp; 이러한 'Reliability'는 다음의 다섯 가지 요소로 결정된다.
**=> Readability & Writability, Type Checking, Exception Handling, Aliasing**

<br>

**(1) Readability & Writability**
&nbsp; &nbsp; 가독성과 작성능력은 간접적으로 '신뢰성'에 영향을 준다. **익숙치 않은 방법이 쓰인 코드를 보면 우리는 '오류가 있을 것'이란 예상을 할 수 있기 때문**이다.

<br>

**(2) Type Checking**
&nbsp; &nbsp; **'Type Compatibility(자료형 일치)'를 체크하는 기능이란, 두 개 이상의 변수끼리, 혹은 변수와 상수끼리의 대응 관계가 적절한지를 보는 기능**이다. 이러한 기능의 제공은 프로그램의 신뢰성을 높이는 요소이다.

- Static Type Binding과 Dynamic Type Binding에 대해 설명한 바 있다. Dynamic한, 즉, Run-time에 타입 체킹을 하는 것은 마치 동적할당과 같기 때문에 느리다고 설명한 바 있다. 따라서 **Compile-Time에 타입을 체킹하는 Static 방식이 속도 측면에서 우수**하다고 배운 바 있다.

- **C언어의 경우, 통념과 다르게, 완벽하게 Type Checking이 이루어지는것은 아니다. 의외로, 생각해보면 C언어는 Parameter Passing 시 Type Checking이 이루어지지 않는다**. 파라미터 선언은 int로 해놓고, caller에서 char값이나 float값을 넣어도 컴파일에 전혀 문제가 발생하지 않는다.

- **또한, C언어의 경우, array의 시작 주소값 패싱 외에는 배열과 관련된 자세한 정보가 기본적으로는 전달되지 않기 때문에 함수 내에서 전달받은 배열의 크기를 넘어가는 인덱싱이 전혀 감지되지 않는다.**

- **반면에, Pascal의 경우, array에 대한 Range Checking이 수시로 자동으로 수행**된다. 따라서, 예를 들어 Pascal에서 array를 함수의 파라미터로 넘길때, 인덱스가 벗어난 요소를 접근하려고 하면, C언어와 다르게 오류를 감지해 알려주는 기능이 존재한다.

**~> 즉, Parameter Passing 시의 Type Checking 측면에서는 Pascal이 C보다 Reliable한 것**이다. 물론, 그 기능을 위해서 Pascal은 Run-time Range Checking이 이루어지기 때문에 속도가 상대적으로 **C보다 느린 측면**이 있지만 말이다. 

**~>** C언어 개발자가 이러한 기능을 몰라서 안넣은 것이 아니라, 속도 향상을 위해 일부러 안넣은 것이다.

_=> 결론적으로, Type Checking 기능은 프로그램의 Reliability를 향상시키는 요소이다._

<br>

**(3) Exception Handling (예외 처리)**
&nbsp; &nbsp; **Run-Time Error나 비정상 상황을 잡아내 대응하는 능력은 프로그래밍 언어, 프로그램의 신뢰성을 높이는 요소**이다.

**ex) 앞서 예로 든 Division by Zero Error 감지 기능**이 바로 이러한 예시이다. Exception Handling에 관한 자세한 이론적 개념은 SP 포스팅에서 하도록 하겠다.

<br>

**(4) Aliasing**
&nbsp; &nbsp; 본디 Aliasing이라 함은, 컴퓨터 그래픽에서 발생하는 계단현상과 관련한 개념인데, Computer Science에서의 Aliasing은 다음과 같은 개념을 칭한다.

> _**Aliasing in CS : 어떤 메모리 셀을 Access할 수 있는 방법이 두 가지 이상일 때, 그러한 기능을 Aliasing이라 한다.**_
(Aliasing : Two distinct ways of referencing for the same memory cell)

**ex) C언어에서 일반 변수와 포인터 변수로 각각 같은 메모리 셀을 접근할 수 있음을 떠올리면 된다.**

```c
int a = 3;
int *ptr = &a;		// *ptr과 a는 같은 메모리 셀에 위치한 Value를 가리킨다.
```

> **이러한 Aliasing은 프로그래밍 언어의 Reliability를 떨어뜨리는 요소이다.** 프로그래머가 실수할 가능성을 높여주는 요소이고, 또한, **디버깅 난이도를 높이는 요소**이기 때문이다.

**ex)** C나 Pascal의 포인터 변수, FORTRAN의 Equivalenced Variable 등이 바로 이러한 예시에 해당한다.

<br>

### 4. Cost (비용)
&nbsp; 프로그래밍 언어를 평가하는 맥락에서 Cost라 함은 다음과 같은 요소들을 일컫는다.

```
- 프로그래머에게 언어를 가르치는데 드는 Cost
- 언어로 프로그램을 작성하는데 드는 Cost
- 언어의 컴파일링 프로그램의 Cost
- 언어로 작성된 프로그램의 수행 Cost
- 프로그램을 유지보수하는데 드는 Cost
```

&nbsp; 실제 현금 비용이 아니라(물론 넓게 보면 이것도 포함임), 큰 맥락에서 효율을 의미한다.

&nbsp; 이러한 'Cost'는 다음의 다섯 가지 요소로 결정된다(상기한 Cost 분류).
**=> Programmer Training, Program Writing, Compilation, Execution, Maintenance**

<br>

### 5. 기타 Criteria
&nbsp; 이밖에도, 프로그래밍 언어를 평가하는 기준으로 다음과 같은 3요소가 더 있다.

- **Portability (이식성) : "얼마나 많은 Platform에 해당 언어의 컴파일러가 존재하느냐?"**를 기준으로 한다. C언어의 경우 이 방면에서 최고로 평가받는다.

- **Generality (일반성) : "얼마나 많이, 다양하게 응용될 수 있느냐?"**를 기준으로 한다.

- **Well-Definedness (문서화 정도) : "언어에 대한 문서화가 얼마나 잘 되어 있느냐? Document 기능이 잘 갖추어져 있느냐?"**를 기준으로 한다.

**=>** 이처럼, 다양한 Criteria들이 있는데, 이중에서 'Support for Abstraction'이 가장 핵심적인 요소이다. 이에 대한 내용을 잘 기억하도록 하자.

<br>

**여담)** 라이브러리를 많이 가져와 쓰는 것은 언어 외적으로 언어의 용도와 별개의(아닐수도 있지만 대체적으로!) 기능을 추가해 쓰는 것이므로, 이는 "언어의 평가 기준"으로 두지 않는 경우가 많다.

<br>

**※ Buffer와 Cache의 비교**
- 공통점 : **둘 다 저장장치**이다.
- 차이점
  - **Cache는 자주 쓰일 것으로 예측되는 data들을 상대적으로 가까이(Access 속도가 빠른 위치에) 가져다두기 위해 있는 메모리 공간**이다. 사실 그래서 캐시라 함은, 특정 장치를 가리키는 용어가 아니라, 이러한 목적을 가진 메모리 공간을 모두 통칭하는 용어이다.
  **ex) 가장 대표적인 비유로는, 책상 옆에 있는 책장과 책상 위에 있는 책꽂이**이다. 책꽂이가 캐시, 책장이 메모리인 것이다.
  **ex) 캐시는 상대적**이다. 메인 메모리는 디스크(SSD)에 비해서는 '캐시'인 것이다.
  **ex)** 메인 메모리에 비해서는 레지스터나 캐시메모리가 캐시인 것이다.
  
  <br>
  
  - **Buffer는 속도가 다른 저장장치 사이에서 완충 작용을 위해 사용되는 메모리**를 일컫는다.
  **ex)** 네트워크를 통해 들어오는 유튜브 영상물의 처리 속도는 불규칙적이다. 그에 반해 핸드폰의 영상 실행 속도는 규칙적이다. 그래서 그 사이에 버퍼를 두어 속도를 맞추는데, 이를 **'버퍼링'**이라 하는 것이다.
  **ex)** 버퍼 사이즈는 클수록 좋다. 하지만, 너무 크면 오히려 좋지 않다. 버퍼에서 정보를 끌어오는데 시간이 걸리기 때문이다. **적절한 수준에서 용량이 커야 좋다.**
