## 언어란 무엇인가?
&nbsp; 본디 '언어'란, 다양한 정의가 가능하겠지만, "인간 사이의 소통을 위한 수단, 의사 전달 수단"이라고 간단하게 정의할 수 있다. 바로 이러한 정의에서 우리는 '**프로그래밍 언어**'도 정의할 수 있다.

> 프로그래밍 언어란, 프로그래머와 컴퓨터가 소통하기 위한 수단이다.
"**프로그래머의 생각을 컴퓨터에 전달하기 위한 수단**"이라고 할 수 있다.

&nbsp; 인간이 사용하는 자연어로는 컴퓨터와 소통할 수 없기 때문에 기계어를 참고해 일정한 규칙을 만들어서 특정한 "언어"를 만들어낸 것이다.

&nbsp; **현대의 대부분의 고급 프로그래밍 언어는 "Von Neumann(폰 노이만) 컴퓨팅 모델 구조"를 바탕으로 두고 설계**되었다. 이때, 폰 노이만 컴퓨팅 모델이란, 가장 간단하게 설명하면 아래와 같다.

> **Von Neumann Architecture** : 아래의 그림과 같이, 특정한 블록에 CPU와 메모리가 있고, CPU 내에 ALU와 Register 등이 있으며, 그러한 블록과 연결된 입출력 장치가 있는, **우리가 일반적으로 인지하고 있는 컴퓨터 모델 구조**를 의미한다.

![](https://images.velog.io/images/junttang/post/1414d922-816c-4e5a-b667-10645e018d33/1024px-Von_Neumann_Architecture.svg.png)
**(폰 노이만 구조 = Input -> [ CPU with ALU & Memory + Memory ] -> Output)**


<br>

&nbsp; 즉, 고급 프로그래밍 언어 **HLL은 다양**하지만, 결국 이들은 **하나같이 '폰 노이만 구조'를 바탕**으로 설계되었기 때문에 _**일련의 공통적인 원리와 규칙**_을 가지고 있다. 본인이 배우는 '프로그래밍 언어'란, 특정한 프로그래밍 언어를 다루는 것이 아니라, 다양한 **프로그래밍 언어 전반에 걸쳐 나타나는 공통적인 특징과 원리, 배경에 대해 다룬다.**

&nbsp; 그렇다면, 이러한 프로그래밍 언어의 특징을 알아야 하는 이유는 무엇일까?

>_ **WHY STUDY?**_
언어의 원리를 이해하면, 내가 짠 프로그램이 컴퓨터 **하드웨어에서 어떻게 돌아가는지를 미리 예측**할 수 있기 때문에 훨씬 더 수행속도가 **빠른 프로그램**을 작성할 수 있게 된다.

> _**WHY STUDY?**_
또한, **새로운 언어**를 배우기가 무척 수월해진다.

&nbsp; 그렇다. 프로그래밍 언어의 특징을 잘 이해하고 있는 사람은 보다 더 효율적인 프로그램을 짤 수 있고, 더불어 새언어에 대한 적응력도 높일 수 있는 것이다.

<br>

## 프로그래밍 언어의 분류
&nbsp; 프로그래밍 언어는 다양한 기준으로 나눌 수 있지만, 가장 큰 범위로 분류하면 아래와 같이 나눌 수 있다. (이러한 내용을 총망라하여 "프로그래밍 패러다임"이라고 부른다.)

- **Imperative Language(명령형 언어)** : **절차형과 객체지향형**을 포함해 우리가 흔히 다루는 **'폰 노이만 구조' 바탕의 언어들**이 이에 포함된다. 즉, **하드웨어를 고려하여 개발**된 언어라고 봐도 된다. C, Pascal, Python, Java 등

- **Declarative Language(선언형 언어)** : 명령형 언어의 반대 급부 개념으로, **하드웨어와 상관없이 만들어진 언어**로, 가장 간단한 정의로, '알고리즘을 명시하지 않고 **목표만을 명시**하는 프로그래밍 방식'을 지원하는 언어라고 보면 된다.

   - **선언형 언어의 특징** : 개발 자체가 하드웨어를 고려치 않고 이루어졌기 때문에 **하드웨어로 프로그램을 맵핑시키는 과정이 복잡**하고 어렵다. 따라서 Declarative Language는 **필연적으로 인터프리터(가상 머신)를 이용**해 구동된다.
**~>** 따라서, 인터프리터 방식을 택하기 때문에 선언형 언어는 Imperative 언어에 비해 수행 속도가 **느린 특징**이 있다. Imperative 언어는 하드웨어와 밀접하기 때문에 수행속도가 Declarative보다 빠르다.

<br>

&nbsp; **여담)** 이때, 오해하지 말아야할 것이, **"컴파일러 vs 인터프리터"의 사용 여부**는 결코 Imperative 언어와 Declarative 언어의 구분 기준이 아니라는 점이다. 인터프리터와 컴파일러는 단순히 프로그램을 동작시키기 위한 방법의 차이일 뿐이고, 이들의 사용 여부는 **언어의 종류에 독립적**이다.
**~>** 일례로, C언어의 경우 인터프리터로 충분히 구동이 가능하다.
**~>** 또 다른 예시로, **Basic이나 Python과 같은 일부 Imperative 언어는 컴파일러 대신 인터프리터를 사용**한다. 학습용 언어인 Basic과, 직관성을 필두로 하는 Python은 그들의 '편리함'을 살리기 위해 어떤 변수의 타입을 수행 도중 계속 변화시킬 수 있다. 이를 Dynamic Type Binding이라 하는데, 이는 컴파일러로 수행하기 매우 어렵다.

<br>

### ※ Type Binding : 프로그램의 변수와 같은 구성요소의 Type을 지정하는 기능
- **정적 타입 바인딩(Static Type Binding) : 컴파일 시간에 Type을 지정**
ex) C언어 : 컴파일 시간에 변수의 데이터 타입 지정
~> 프로그램의 효율성, 즉, 실행 속도가 빨라진다.

- **동적 타입 바인딩(Dynamic Type Binding) : 실행 시간(Runtime)에 Type을 지정**
ex) Python : Runtime에 변수 값에 따라 변수의 데이터 타입 지정
~> 적용성이 높고, 아무래도 직관적이며 편리함.

**=> Static Type Binding은 주로 컴파일러가, Dynamic Type Binding은 주로 인터프리터가 수행**한다. 변수 타입이 Dynamic하게 변화하는 Python의 경우, Imperative 언어이지만 Interpreter로 수행되며, 프로그램 수행 중간 중간 계속해서 Type 처리가 이루어지므로 프로그래머 관점에서는 편리하지만, 동시에 수행속도는 필연적으로 느려진다.

(컴파일러는 Static Type Binding과 밀접, 인터프리터는 Dynamic Type Binding과 밀접)

<br>

### ※ "Efficiency vs Flexibility" Tradeoff
&nbsp;&nbsp; 상기한 Static & Dynamic Type Binding 문제로 인해 **프로그래밍 언어는 항상 '효율성'과 '융통성' 사이의 Tradeoff를 필수로 한다**. 무슨 말이냐 하면, 효율성을 살리면 융통성이 떨어지고, 융통성을 살리면 효율성이 떨어진다는 것이다. 앞서 이야기한 C와 Python의 효율성 차이가 바로 이러한 점에서 발생한다.

## 마무리
&nbsp; 장황한 개괄은 여기서 마치겠다. 앞으로 연재할 PL(Programming Language) 공부에서는 다음과 같은, 프로그래밍 언어의 공통적 기능들에 대해서 이들의 원리와 배경(WHY)에 대해 다룰 것이다.

- **Data Type**
- **메모리 Addressing 방법** : 학부에서 어셈블리어에 대해 공부할 때 우리는 프로그램이 메모리를 Byte 단위로 접근함을 확인한 바 있다. 물론, Integer가 Word 사이즈로 메모리를 소유하고 있음을 우리는 알고 있지만, 이는 해당 언어가 그렇게 설계된 것이고, 사실 어셈블리 레벨로 파고 들어가보면 Byte 단위로 메모리 어드레싱을 하고 있음을 알 수 있다. 이러한 메모리 어드레싱 기법을 이용해 언어에서 기본 제공하는 자료구조들을 구분한다는 점을 배우게 될 것이다. (ex. Array vs Structure)
- **조건문, 반복문, switch문**
- **Subprogram 개념** (프로시저 vs 함수, 그리고 파라미터 패싱 기법)
- **OOP 개념** (객체, 메시지, 상속, Polymorphism 등)
