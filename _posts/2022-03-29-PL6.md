## 구현 방법론
&nbsp; **다른 보조 수단의 도움이 없다고 했을때, 하드웨어가 알아들을 수 있는 것은 오로지 Machine Language**이다. **기계어보다 Low-Level인 개념은 '마이크로 신호' 정도** 있다.

&nbsp; 하드웨어를 컨트롤하기 위해선 기계어가 필요한데, Naive한 기계어를 좀 더 편하게 알아보기 위해 **기계어와 Text를 1대1로 맵핑한 것이 Assembly Language**이다.

&nbsp; 그리고, 그러한 **어셈블리어 명령 여러개를 조합해서 하나의 고급 명령으로 사용하는 것이 High-Level Language**이다.

> _**특정 소프트웨어가 고급언어 인터페이스를 컴퓨터에 제공하기 때문에 고급언어 명령을 알아듣는 것이다.**_

> _그러한 소프트웨어에는 **Compiler, Pure Interpreter, Hybrid System**이 있다._

- **Compiler** : 고급언어를 기계어로 변환한다.
- **Pure Interpreter** : 순수하게 인터프리팅한다.
- **Hybrid System** : 고급언어의 중간 코드를 만든 후 인터프리팅한다.

<br>

**~> 고급 언어로 구현한 프로그램은 많은 'OS 기능(Operating System Facilities)'을 필요로 하기 때문에, 상기한 소프트웨어가 OS와 협업해, 프로그램이 직접 다이렉트로 프로세서에 접근하지 않게 한다.**

**~> 이 관계성을 '가상 컴퓨터(Virtual Computer)' 상에 계층도로 묘사**하면 아래와 같다.

![](https://images.velog.io/images/junttang/post/60a938f6-a973-41d3-ace4-37052a93d0db/vfdsadfs.png)

<br>

> **C언어로 명령을 내렸는데, 마치 하드웨어가 C언어의 고급 명령을 알아듣고 일을 하는 것 같아 우리는 C 컴파일러를 'C 가상 머신(C Virtual Machine)'이라고 한다.**

- Pascal Compiler도 마찬가지 이유로 **Pascal Virtual Machine**이라 부른다.

- C / Pascal Virtual Machine은 **Interpreter Virtual Machine과는 다른 의미**를 지닌다.
  - **우리는 보통 인터프리터(Interpreter)를 가상머신(Virtual Machine)이라고 부른다.**
    - **ex) LISP Virtual Machine**
    - **ex) Java Virtual Machine** (사실 이 친구는 컴파일 과정이 있긴 하다. 하단 서술)
  
<br>

> **Bare Machine : 하드웨어**를 의미한다. 하드웨어는 기계어를 알아듣고, 그 내부에서 Microinstruction, Microsignal을 이용해 소통한다.

<br>

> **Operating System OS : Bare Machine 위에서 시스템 하드웨어를 관리**할 뿐만 아니라, **응용 소프트웨어를 실행하기 위해 하드웨어 추상화 플랫폼과 공통 시스템 서비스를 제공**하는 **시스템 소프트웨어**이다.

<br>

**여담)** C언어는 컴파일러로만 구현할 수 있는가? : 지난 1.1 포스팅에서 언급했었던 것처럼, 아니다. 컴파일 과정이 적합할 뿐, 인터프리터로도 구동할 수 있다.
**~> Run-Time에 Dynamic하게 일을 많이 하는 언어들이 인터프리터로 구현하기 적합**하다.

<br>

### Compilation
- **프로그램을 컴퓨터에서 직접 수행할 수 있게 기계어로 변환하는 과정**을 컴파일이라 한다.

- 컴파일 변환 과정이 완료되면 **매우 빠르게** 프로그램을 수행할 수 있다.

- **C, COBOL, Ada, Pascal** 등이 여기에 해당한다.

<br>

### Hybrid Implementation System
- **Compiler와 Pure Interpreter의 혼합형**이다.

- **고급 언어로 작성된 프로그램을 '중간 언어(Intermediate Language)'로 변환한 후, 거기서 Interpretation을 수행**한다.

- Java가 여기에 해당한다.
  - Java Virtual Machine : Java로 작성된 프로그램은 1차적으로 컴파일을 거쳐 **Java Byte Code**가 되고, 그 Byte Code에 대해 Interpreter가 동작한다. 그래서 사실, 엄밀히 표현하면 **'Java Byte Code Virtual Machine'**이란 말이 맞다.
  
![](https://images.velog.io/images/junttang/post/90dedf1e-f78a-4922-bbc3-caf19f54f5b3/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

<br>

### Pure Interpretation
- **Interpreter라고 부르는 프로그램을 이용해 '고급 언어로 작성된 프로그램'을 해석(Interpretation)한다.**
  - _변환(Translation)이 아니다._
  
- **Interpreter : 시뮬레이션 소프트웨어, 머신으로, CPU가 Physical Control Flow(Fetch->Decode->Execute->WriteBack)로 기계어 명령을 수행하듯이, 고급언어 명령에 대해 'Fetch->Execute' Cycle을 수행하는 프로그램이다.**

<br>

- **소스 레벨의 디버깅이 쉽다.**
  - **모든 Run-Time Error Message는 소스 레벨에서 일어나기 때문**이다.
  
- 컴파일러로 만들어진 프로그램에 비해 **속도가 현저히 느리다.**

- **LISP, APL, Prolog, Python** 등이 여기에 해당한다.

![](https://images.velog.io/images/junttang/post/d16a6b31-938a-4519-9d07-2823cbfd1ee1/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C%20-%20%EB%B3%B5%EC%82%AC%EB%B3%B8.png)

<br>

### Cross Compilation
- 컴파일러가 A라는 프로그래밍 언어로 작성된 소스 코드를 받아, 현재 컴파일러가 위치한 하드웨어가 아니라, **다른 하드웨어를 위한 Machine Code를 만들어낼 때, 이러한 행위를 'Cross Compilation'이라고 부른다.**
  - 개발된지 얼마 되지 않은 **'초창기 하드웨어'**에 적절한 컴파일러가 아직 개발되지 않았을 때, 이러한 크로스 컴파일 과정을 수행하는 경우가 종종 있다.
  
<br>

### Preprocessor
- **'전처리기'라고 하며, 컴파일 전에 미리 전처리문을 처리(변환)하는 소프트웨**어를 칭한다.

- **'매크로 확장기(Macro Expander)'나 C언어의 전처리기**가 대표적이다.

**ex)** #define SUM(A,B) A+B

<br>


> **컴파일러는 사전에 한 번에 모두 기계어로 변환해 수행을 시작하는 것이고, 인터프리터는 실시간으로 한 문장씩 받아들이면서 수행하는 것**이다.
>> _인터프리터는 하나 하나 동작하는 매커니즘을 소프트웨어로 수행한다._
>>> 인터프리터 방식의 고급 언어 명령을 받으면, 그 명령을 특정 소프트웨어로 **미리 할당해놓은 메모리 공간에 변수를 만들어 실시간으로 처리**한다. 물론, 당연히 결국엔 기계어로 변환되지만 말이다.

**ex) 예를 들어, 미리 Byte Array를 사이즈 1,000,000만큼 할당**해놓고, HLL 소스 코드 'a = b + 1'이라는 명령을 소프트웨어가 받으면, 'array[1000] = array[6000] + 1', 이런식으로 수행한다는 것이다. (인터프리터)

<br>

## 통합 개발 환경(Integrated Development Environment, IDE)
- **소프트웨어 개발을 위한 도구들을 모아놓은 프로그램 환경**을 IDE라고 한다.
  - **File System, Editor, Linker, Compiler, Debugger, User-Interface 등을 제공**한다.
    - **ex)** Microsoft Visual Studio, Eclips, Android Studio 등

- **Syntax Directed Editor** : Syntax를 미리 에디터 레벨에서 알려주고, 점검하고, 알아보기 쉽게 다양한 색으로 표현하고, 등등 **Syntax 지향적인 편집기**를 의미한다. 많은 IDE가 이러한 Editor를 지닌다.
