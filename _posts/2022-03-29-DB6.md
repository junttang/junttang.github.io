## Weak Entity Set
![](https://images.velog.io/images/junttang/post/354f33ee-0100-4238-bcde-068ab0f49b84/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

&nbsp; 오늘 계속 예시로 다룰 **'University Database'의 Schema Diagram(Relational Model)**이다.

- **section Entity는 course_id, semester, year, sec_id를 이용해 유니크하게 식별**할 수 있다.

- 한편, **section Entity와 course Entity는 서로 공통적으로 course_id라는 속성을 가지므로, 이 두 Entity Set이 관계가 있다고 표현**할 수 있다.
  - 이 두 Entity Set 사이에 **'sec_course'라는 Relationship Set**을 만들어보자.

> **단순히 Relationship Set만 추가하면, Data Redundancy가 발생한다. 왜냐? course_id가 section에도, course에도, sec_course에도 나타나기 때문!**

**~>** 당연히, sec_course Relationship Set을 만들지 않으면 데이터 중복성 문제가 발생하지 않는다.
**=>** 하지만, 두 Entity Set의 관계를 ER모델에서 표현할 때, Relationship Set 없이 표현하는 것은 바람직하지 않다.

**~>** 또 다른 방법으로는, section Entity Set에서 course_id 속성을 없애는 것도 방법이다.
**=> 하지만, section의 Primary Key 구성 요소인 course_id를 마냥 없애면, section을 유니크하게 구별할 수 없게 된다.**

<br>

> **Weak Entity Set : Entity Set인데, 그 존재성이 다른 Entity에 의존적인 Entity Set**을 말한다.
>> **그 의존의 대상이 되는 Entity Set은 'Identifying Entity'**라고 한다.

- **Weak Entity Set이 아닌 일반적인 Entity Set을 'Strong Entity Set'**이라고 부른다.

- **모든 Weak Entity Set은 Identifying Entity와 관계**된다. 
  - _**이러한 의존 관계를 영문으로, 'Existence Dependent'라고 한다.**_

- **Identifying Entity Set이 Weak Entity Set을 소유(Own)한다고 표현**한다.

- **Identifying Entity Set과 Weak Entity Set 사이의 관계를 'Identifying Relationship'이라 한다.**
  - 'Identifying Relationship'은 테두리가 두 개인 다이아몬드, **Double Diamond로 표현**한다.

- ER 다이어그램에서, **Weak Entity Set은** 테두리가 두 개인 직사각형, **Double Rectangle로 표현**한다.
  - **Weak Entity Set의 Discriminator들은 밑줄(Dashed Line)을 긋는다.**
    - **Discriminator : Weak Entity Set의 Primary Key**를 의미한다.

![](https://images.velog.io/images/junttang/post/974fe511-4382-4db4-a72f-bc4f6eb97ddd/bgrfeds.png)

**=>** 이게 무슨 말인지 요약적으로 설명하겠다. **course와 section이라는 Entity Set이 있는데, 둘 다 설계하고 보니 course_id가 들어가고, 서로 관계가 있더라**. 그리고 단순히 관계가 있는게 아니라, **section을 정의하고 구분하는데에 course_id가 핵심적인 요소더라**. course_id가 course의 핵심인 것은 당연지사. 

**=>** **그래서 sec_course라는 이름의 관계를 그려주려는데, 양쪽에 course_id가 있으니 이는 불필요한 데이터 중복**이다. 이때, **section은 course의 course_id가 없으면 안되는 친구(Primary Key의 일부)고, course_id는 course에서 유일한 기본키이므로 없애면 안된다.**

**=>** 따라서, **section의 course_id를 ER모델 상에서 잠시 가리고, 대신, section한테 course_id가 중요한 친구라는 것을 까먹지 않게, 이를 Relationship Set의 형태로, 그려놓자. **또한, 다른 관계집합과 구분을 위해 **더블 다이아몬드를 그려주고, 의존적인 section에게도 더블 테두리를 해주자.**

<br>

- 위와 같은 사고를 거쳐 ER모델에서는 이러한 상황을 위와 같이 표현하는 것이다.
  - **section의 Logical Schema에는 여전히 course_id가 있음**에 주목하자.

- **Weak Entity Set은 Identifying Entity Set에 의존적이므로, IES가 없으면 존재할 수 없다. **
  - 따라서 **Weak Entity Set은 Identifying Relationship Set에서 Total Participation의 형태를 띈다.**
    - **그래서 연결선도 Double Line**인 것이다! ★

<br>

## Redundant Attribute
&nbsp; 다음과 같은 Entity Set과 각 Set의 Attributes가 있다고 해보자.

- **student : ID, name, tot_cred, dept_name**
- **department : dept_name, building, budget**

&nbsp; 이 두 Entity Set은 stud_dept라는 Relationship Set을 가지고, 관계는 Many(student)-to-One(department)이다.

![](https://images.velog.io/images/junttang/post/2c1701f1-8f13-41b6-8b72-6f47053f8fe9/bgrfedws.png)

**~> dept_name이 양쪽에서 모두 나타나고 있다. (Data Redundancy)**

> Relational Model에서는 상관없었겠지만, **ER 모델에서는 이러한 데이터의 중복성은 허용치 않는다.**
>> **이 상황에선, student의 dept_name을 지우는 것이 현명하리라. department에서 dept_name은 기본키이므로. 의미상.**
>>> **단, 이때, dept_name은 student의 기본키일리 없다. 의미상. 따라서, 바로 위에서 이야기한 Weak Entity Set에 해당되지 않는다.** _그냥 Redundant Attribute인 것이다._

<br>

> _**ER Model에서는 Redundant Attribute를 Relationship Set으로 변환한다. 단, 의존성을 형성하는 Attribute일 경우, Weak Entity Set 표시를 하는 것이다.**_

<br>

## ER Model to Relational Model (Reduction)
### Strong & Weak Entity Set Reduction
- **ER Model의 Entity Set과 Relationship Set들을 특정한 규칙에 의거해 DB Relational Model의 Relation Schema로 변환**할 수 있다.
  - "E-R Diagram to a Collection of Schemas"

> **Relationship Set의 Reduction은 기본적으로 별도의 Relation Schema로 만든다는 것이다.** ★
>> **MtoM** : 양쪽의 기본키를 모두 기본키로 하는 Relation을 만들어 연결한다.
>> **OtoM / MtoO** : 양쪽의 기본키를 모두 가져오되, Many쪽의 기본키만 기본키로 취급하는 Relation을 만들어 연결한다.
>> **OtoO** : 둘 중 하나의 Entity Set에 반대편의 기본키를 Extra Attribute로 넣어준다.
>>> **OtoM / MtoO**의 경우, Many 방향의 참여도가 Total이면, 다른 처리가 있다. (하단 서술)

- **Attribute는 그대로** 넘어간다. Column 역할을 하는 것이다.

- **Strong Entity Set의 경우 그대로 딱히 바뀌는거 없이 Relation이 된다**.

- **Weak Entity Set의 경우, 자신의 Attribute들에다가, Identifying (Strong) Entity Set의 Primary Key를 모두 포함**해온다. ★

![](https://images.velog.io/images/junttang/post/2f1e2c41-eeae-4464-ad8c-58325454479e/thrgefd.png)

**~> course( course_id, title, credit )
~> section( course_id, sec_id, semester, year )**

<br>

### Attributes Reduction
- **ER 모델의 Composite Attribute의 경우, Relational Model로 옮길 때는, 각 Component Attribute들로 쪼개 별도의 Attribute들로 만든다.**
  - Component들의 이름은, 원래 원칙상으로는, **"(Compostie)\_(Component)"의 형태**로 작성해야하지만, Component들의 이름이 딱히 **모호성 없이 구분된다면, 그냥 "(Component)"로** 간다.

- **Multivalued Attribute의 경우, Relational Model로 옮길때, 속성에서는 제거한 후, 별도의 Schema를 따로 만들어준다.** 이때, Multivalued Attribute가 _**속했던 Entity Set의 Primary Key를 받아오며, 동시에 Multivlaued였던 Attribute도 Primary Key에 포함**_시킨다.

![](https://images.velog.io/images/junttang/post/74f0b785-fdcf-4675-9a3c-f6c17dcb1506/ntbgrfds.png)

**~>** ER Model에서 Multivalued였던 phone_number Attribute를 별도의 Schema로 뽑아내고, 기존 Entity Set의 Primary Key를 물려받았다.
**=>** 이렇게 하면 좋은 것이, 예를 들어, 교수ID 22222인 교수가 투폰을 쓰고, 각 번호가 456-7890, 123-4567이라면,
**--->** (22222, 456-7890), (22222, 123-4567)이란 Tuple을 서로 다르게 잘 취급할 수 있고, 동시에 instructor Schema와 관계를 맺어, 필요할 때 뽑아 쓸 수 있음.

![](https://images.velog.io/images/junttang/post/c893a25f-98ac-4dd8-9937-05c37f1821dc/nhbgdfs.png)

**~>** time_slot Entity Set에서, time_slot_id는 Primary Key이고, 그 다음의 day, start_time, end_time은 모두 {}안에 있으므로 Multivalued Attributes이다.
**=>** 따라서, Multivalued는 별도로 뺄 것이므로, 제거해놓고 Schema로 전환하면, time_slot( time_slot_id )이다.

**~>** 한편, 별도로 만든 Multivalued의 Schema는 day_time( time_slot_id, day, start_time, end_time )이다. 속해있던 Entity Set의 Primary Key를 물려받은 것을 알 수 있다.
**=>** 그런데, 만약 **이렇게 되면, time_slot Schema가 쓸모가 딱히 없어지고, 괜히 중간 단계만 늘어난 모양새가 된다.**
**~~>** 따라서, 이럴땐, time_slot( time_slot_id, day, start_time, end_time )으로, 단일의 schema로 둘을 하나로 만든다.

> _**Entity Set에서 Primary Key를 제외한 모든 속성이 Multivalued이면, Schema 전환 시, 그냥 전환해도 문제가 없다.**_

<br>

### Many-to-Many Reduction
![](https://images.velog.io/images/junttang/post/261e1aff-21dd-4401-a2a8-21b7b7871de6/myuthrged.png)

**~> 위와 같이, Many-to-Many Relationship Set의 경우, 별도의 Schema를 만들고, 그 Schema의 기본키로는 두 Entity Set의 기본키들을 가져와 쓴다**.
**=>** 위와 같이, **가져온 기본키가 이름이 같은 경우, "(EntitySet)\_(Attribute)"** 이런식으로 표기한다.

<br>

### Redundancy of Schemas
- **MtoO / OtoM 관계의 Relationship Set에서, Many 쪽이 Total Participation인 경우 :  Relation을 따로 만들지 않고 아래와 같이 처리한다.**
  - _**One 쪽의 Primary Key를 끌어와 Many 쪽의 일반 Attribute(Not Primary Key)로 넣어준다.**_
  
  - _Many 쪽이 Total이 아닌 경우엔, 이렇게 Extra Attribute를 넣어주면, NULL값이 들어올 가능성이 있다._

- **OtoO 관계의 Relationship Set에서는, 둘 중 하나의 Entity Set에 반대편의 Entity Set의 Primary Key를 끌어와 일반 Attribute로 넣어준다.**

![](https://images.velog.io/images/junttang/post/c99da11c-b8ed-4b07-95be-4bcdd6fb23d6/kumjynhtgbrfvd.png)

**~> instructor나 student 입장에서는 dept_name Attribute가 Foreign Key가 되는 것**이다. **department라는 Relation의 Primary Key이므로!**
