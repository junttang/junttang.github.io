&nbsp; 지난 1.1 포스팅에 이어지는 내용이다. **프로그래밍 언어의 공통된 특징**, 그중에서도 가장 기초적인 특성들에 대해 개괄적인 소개를 하는 시간이다. 아직 세부 내용에 들어가는 것은 아니다.

### 특징1. Compiler is Hardware Dependent!

&nbsp; 아래와 같은 C언어 소스 코드가 있다.

```c
int main(void) {
	int i, j;
	float f = 0.0;

	for (i = 1; i <= 100; i++) {
		f = f + i;		 // Warning! -> Coercion
		f = f + f;		// Type Casting은 명시적 변환, Coercion은 자동 형변환
	}
    
	i = 30; j = 20;
	swap(i, j);
    
    return 0;
}

void swap (a, b) {		// Call by Value ~> 메인의 i, j는 swap함수에 의해 바뀌지 않음.
	int temp;
	temp = a;
	a = b;
	b = temp;
}
```

&nbsp; 이러한 소스 코드는 1.1포스팅에서 언급했던 것처럼, **Von-Neumann Computing Model**에서 아래와 같은 흐름으로 돌아간다. (C 기준)

> **main.c -> 컴파일 -> main.obj -> 어셈블&링크 -> main.exe -> 로드 -> 메모리 적재 및 실행 준비**

![](https://images.velog.io/images/junttang/post/7ccc6f4d-4656-4f83-ad9d-92deb1854c5c/%EC%A0%9C%EB%AA%A9%20%EC%97%86%EC%9D%8C.png)

&nbsp; 이때, **컴파일러는 항상 Hardware Dependent**하다. 즉, 하드웨어라 함은, CPU를 필두로 보는데, **CPU가 몇 비트 CPU냐에 따라 컴파일러의 구성도 달라진다.** 16비트 CPU, 32비트 CPU, 64비트 CPU 등으로 구분되는데, 이때, _**CPU의 비트수에 따라, 그 CPU에 조응하는 컴파일러가 취급하는 Word의 사이즈가 달라진다.**_

> _**일반적으로, N-Bit CPU에서 돌아가는 컴파일러는 Word 사이즈를 N-Bit로 취급한다.**_

&nbsp; 따라서, 16비트 CPU에서는 Word가 16Bit = 2Bytes를 의미한다. 만약, 16비트 CPU에서 아래와 같은 코드를 돌리면 프로그램은 정상적인 결과를 내지 못한다.

```c
int i = 65536;
```

&nbsp; 16비트 CPU에서 Word의 사이즈는 16Bit이고, Word라 함은 Integer의 사이즈를 의미하는데, 16비트 CPU에서는 최대 16Bit까지만 인식 가능해 65536 = 2^16이란 값은 변수에 할당할 수 없다. 쓰레기 값을 가지게 된다. (Overflow)

&nbsp; 아무튼, 하고 싶은 말은, **프로그래밍 언어에서 컴파일할 때는, CPU에서 다루는 최대 비트 사이즈가 몇이냐는 프로그램에서 취급하는 최대 정수 크기, 메모리 할당 사이즈를 결정한다**는 점을 말하고 싶은 것이다.

> _Compiler is always Hardware Dependent !_

### 특징2. Loop의 동작 방식

&nbsp; 한편, 프로그래밍 언어의 또 다른 특성으로는, 루프의 동작 방식이다.

- **Pre_Test 방식** : 말그대로, 반복을 시작할 때, **반복의 '조건' 검사를 Pre, 매 Iteration의 시작 전에 한다**는 것이다. C언어의 for문이 바로 여기에 해당한다.

- **Post_Test 방식 **: 말그대로, 반복을 시작할 때, 반복의 **'조건' 검사를 Post, 매 Iteration이 끝나고 한다**는 것이다. C언어의 do ~ while문이 바로 여기에 해당한다.

**~> **각 프로그래밍 언어들은 모두 이러한 루틴의 반복문을 각기 다른 방식으로 채택한다.

<br>

### 특징3. Inconsistent Type 처리

**※ 변수의 Type을 선언하는 것의 의미**

&nbsp; 어셈블리어는 기계어와 Text를 대응시키는 언어이다. 이 어셈블리어는, 학부 수업 때 배웠다시피 Typeless하다. HLL이라고 항상 Type이 있는가? 그것도 아니다. Python의 경우 Typeless하다. 하지만 반대로 C언어는 Type을 반드시 선언해주어야 한다.

**~> Type 선언(or Checking)은 왜 하는 것일까? **: 프로그래머에게 소스 코드 내에 **Language Rule에 어긋나는 정보가 있는지를 미리 체크**해서 _**'에러 유무'**_를 알리기 위함이다. 나아가 에러를 없앨 수 있게 도움을 주는 목적이기도 하다.
**~> 미리 Type을 선언함으로써 컴파일러가 변수의 저장공간을 Runtime 이전에 알맞게 계산해서 할당**해주게 된다. 결국, 이전 포스팅에서 언급한 Static Type Binding으로, 미리 Type을 선언함으로써 _**'속도의 향상'**_도 도모할 수 있다.

> **Type 선언의 장점 : 컴파일러를 통해 문법 어긋남과 에러 체크 가능, 나아가 프로그램 수행 속도 향상!**

**※ Dynamic Type Binding이 느린 이유** : C에서 malloc이나 free같은 동적할당이 속도가 느린 이유와 유사하다. 
- 미지의 메모리 공간을 Searching하여 적절한 위치를 찾아야함. (**탐색 시간**)
- 해당 위치의 주소값을 복사해 반환해주어야 함. (**메모리 접근 및 복사 및 반환**)
- **Dynamic Type Binding 언어일 경우, 모든 변수에 대해 Runtime동안 사실상의 동적할당이 이루어지고 있는 것**이고, 그래서 느린 것

**=>** Static Type Binding에 비해 부가적인 행위가 많음. 그래서 느림.

<br>

&nbsp; 아무튼, **Static Type Binding 언어의 경우, Type Error Checking**이 이루어지는데, 이는 다음과 같은 두 상황으로 처리된다.
**1) Compile Error** : 아예 컴파일이 안되는 상황
**2) (Warning) Coercion** : **자동 형변환**(Coercion 시 Warning 발생. **컴파일은 된다**.)

**=>** 위의 예시 소스 코드에서, '실수 + 정수'같은 것이 Coercion 상황이고, Compile Error는 아예 자료형을 이상하게 선언하거나, 코드 내에서 조응되지 않는 상황 등

> _프로그래밍 언어 중 Static Type Binding을 채택한 언어의 경우, Type Error Checking이 수행된다. _

<br>

### 특징4. Operator Overloading
&nbsp; **"더하기(+)도 다 같은 더하기 심볼이 아니다."**

&nbsp; Integer Add와 Floating Point Add는 **소스 코드에서 같은 더하기 심볼을 사용하지만, 사실 어셈블리어로 변환해보면 다른 명령(기계어)을 사용**한다. 사실, 알다시피 두 '더하기 연산'은 사용하는 HW도 다르다(ALU와 FPU). 하지만 이들은 HLL 상에서 같은 '+' 기호로 수행된다.

&nbsp; 이것이 바로 프로그래밍 언어들의 공통된 특징 중 하나이다. 이를 **'Operator Overloading'**이라 한다. _**동일한 심볼의 연산자를 상황에 따라 다른 기계어 명령으로 맵핑**_하는 것을 의미한다.

&nbsp; 왜 이런 기능이 있을까? 그렇다. 우리의 일상 생활, **직관과 조응시키고자 함**이다.

**ex)** 대부분의 사칙 연산 기호(+ - * /)는 모두 연산자 오버로딩이 적용된다.

**ex)** 문자열 Concatenation에 쓰이는 '+'도 Operator Overloading이다.

**ex)** 아스타리스크도 Operator Overloading이다. 곱하기와 Dereferencing(Pointing)의 역할을 모두 수행하기 때문이다.

**ex)** 앰퍼센드도 연산자 오버로딩이다. Bitwise AND와 Referencing이 가능하기 때문이다.

<br>

**※ 연산자 오버로딩이 항상 직관적인 것은 아니다.**

&nbsp; 예를 들어 보자. 아래의 코드를 보자.

```c
int a; float b;

a = 5;
b = (a / 2);
```

&nbsp; 이때, b의 값은 무엇일까? 2? 2.5?
=> a/2는 a가 Integer이므로 Integer Division이 이루어진다. 따라서 값이 2이다.
=> b는 float이기 때문에 b = 2;는 Coercion이 이루어져 2.0이라는 값으로 들어간다.
~> 이처럼, 항상 직관과 매칭되진 않는다.

<br>

**※ Pointer Arithmetic**

&nbsp; C언어는 Pointer를 다루는 극소수의 언어 중 하나인데, 이 때문에 Pointer Arithmetic이라는 독특한 기능을 제공한다. 아래의 코드를 보자.

```c
char *cptr, c;			int *iptr, i;
cptr = &c;				iptr = &i;
cptr = cptr - 1;		iptr = iptr - 1;
```

&nbsp; 이때, c와 i의 메모리 주소가 1000이라 하면, cptr은 999로 가고, iptr은 996으로 간다. **포인터 변수가 기억하고 있는 메모리 공간의 크기(변수의 자료형)에 따라 +나 -의 기호가 바이트수만큼의 메모리 공간 이동을 수행**하는 것이다.

> _**Pointer Arithmetic은 C언어만이 가지는 Low-Level한 특징이다. **_

&nbsp; C가 이러한 Low-Level한 특징을 보이는 이유는 **C언어의 용도를 알면 이해**할 수 있다. C언어는 처음 **개발 목적이 시스템 SW, 그중에 유닉스OS를 설계하기 위해 개발**되었음을 배운바 있다. 언어의 용도와 목적 자체가 시스템 프로그램, HW를 만드는 것이기 때문에 포인터라는 개념이 등장하는 것이다.

(메모리 관련 작업이 가능하다고 해서 C언어가 좋은 언어인 것은 아니다. 현대에는 컴퓨터의 수행속도가 과거보다 월등히 향상되었기 때문에 Low Level Familiar Language들의 수행속도 장점이 사실 그다지 중요치 않다. **오히려 '가독성(Readability)'이 훨씬 더 중요시된다. 가독성은 곧 Maintenance와 직결**되기 때문이다.)

<br>

### 특징5. Parameter Passing
&nbsp; 현대 프로그래밍 언어들은 파라미터 패싱을 크게 **'Pass by Value', 'Pass by Reference', 'Pass by Name'**의 방식을 취한다.

**~> By Value** : Caller에서 Callee로만 단반향으로 정보가 넘어간다.
**~> By Reference** : Caller에서 Callee로 넘어간 후, 다시 Callee에서 Caller로 데이터가 넘어온다.

**ex)** 알다시피 C는 Call by Value와 Call by Reference를 모두 지원한다. By Value가 Default이다.

**ex)** FORTRAN의 경우 기본 패싱 방식이 Call by Reference이다.

<br>

**※ 변수의 Scope와 Lifetime 개념'**
- **Scope** : 변수에 접근할 수 있는 영역
- **Lifetime** : 변수가 유지될 수 있는 시간 범위

**~>** Local Variable의 경우 Scope는 호출된 함수 내부, Lifetime은 그 함수가 호출되고 종료되는 순간까지.
**~>** Global Variable의 경우 Scope는 소스 코드 전체, Lifetime은 프로그램 시작부터 종료까지.

**여담) 사실, 전역변수는 OS의 Local 변수이다.**
